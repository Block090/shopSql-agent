# 历史结果回看与会话删除升级方案

## 一、背景

当前项目已经具备基础查询历史能力：

1. 每次查询会创建或更新一个 `session_id`。
2. 后端会保存用户原始问题、补全后的问题、SQL、结果摘要、状态和时间。
3. 前端左侧可以展示历史会话。
4. 点击历史会话后，可以恢复用户问题和结果摘要。

但是当前历史能力还不完整：

1. **历史结果不能完整回看**
   - 历史表只保存了摘要，例如：
     ```text
     返回 6 行，字段：品类、销量、销售额
     ```
   - 没有保存完整表格数据。
   - 用户点击历史会话时，无法看到之前查询出来的表格。

2. **历史会话不能删除**
   - 左侧历史会话会越来越多。
   - 之前后端异常时留下的空会话也无法清理。
   - 用户无法管理自己的会话列表。

因此需要继续升级两个能力：

1. 历史结果可回看表格。
2. 历史会话可删除。

## 二、目标

本次升级完成后，系统应该支持：

1. 查询成功时，后端保存表格结果数据。
2. 点击历史会话时，前端恢复原始表格，而不是只显示摘要。
3. 左侧历史会话支持删除。
4. 删除历史会话时，同时删除该会话下所有查询记录。
5. 如果删除的是当前正在查看的会话，前端自动切到新会话。

## 三、功能一：历史结果可回看表格

### 3.1 当前问题

当前 `query_history` 表大致保存：

| 字段 | 说明 |
| --- | --- |
| `session_id` | 会话 ID |
| `query` | 用户原始问题 |
| `resolved_query` | 上下文补全后的问题 |
| `sql_text` | 生成的 SQL |
| `result_summary` | 结果摘要 |
| `status` | 查询状态 |
| `error_message` | 错误信息 |
| `created_at` | 创建时间 |

这里没有保存真正的结果数据，所以前端只能恢复一句摘要。

### 3.2 升级方案

给 `query_history` 增加一个 JSON 字段：

```sql
ALTER TABLE query_history
ADD COLUMN result_data JSON NULL COMMENT '查询结果数据，用于历史回看表格';
```

新建库时，也要在初始化 SQL 中包含：

```sql
CREATE TABLE query_history
(
    id             BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '历史记录 ID',
    session_id     VARCHAR(64) NOT NULL COMMENT '会话 ID',
    query          TEXT NOT NULL COMMENT '用户原始问题',
    resolved_query TEXT NULL COMMENT '结合上下文后的完整问题',
    sql_text       TEXT NULL COMMENT '本次生成或预览用 SQL',
    result_summary TEXT NULL COMMENT '结果摘要',
    result_data    JSON NULL COMMENT '查询结果数据，用于历史回看表格',
    status         VARCHAR(32) NOT NULL COMMENT '查询状态',
    error_message  TEXT NULL COMMENT '错误信息',
    created_at     DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    INDEX idx_query_history_session_time (session_id, created_at),
    CONSTRAINT fk_query_history_session
        FOREIGN KEY (session_id) REFERENCES chat_session(session_id)
) COMMENT='问数查询历史表';
```

### 3.3 保存规则

查询成功时：

```json
{
  "type": "result",
  "data": [
    {
      "品类": "手机",
      "销量": 24,
      "销售额": 182000
    }
  ]
}
```

后端保存：

| 字段 | 值 |
| --- | --- |
| `result_summary` | `返回 6 行，字段：品类、销量、销售额` |
| `result_data` | 完整 `data` 数组 |
| `status` | `success` |

### 3.4 数据量控制

第一版建议只保存当前查询返回给前端的数据。

因为项目已经有 SQL Guard 要求 `LIMIT`，正常结果不会无限大。

为了更安全，可以增加兜底限制：

```python
MAX_HISTORY_RESULT_ROWS = 100
```

保存时只保留前 100 行：

```python
result_data = data[:MAX_HISTORY_RESULT_ROWS]
```

这样可以避免未来某些查询返回太大时把历史表撑爆。

### 3.5 前端恢复规则

点击历史会话时，后端返回：

```json
{
  "history": [
    {
      "query": "统计 2025 年 3 月各商品品类的销量和销售额",
      "resolved_query": "统计 2025 年 3 月各商品品类的销量和销售额",
      "sql": "SELECT ...",
      "result_summary": "返回 6 行，字段：品类、销量、销售额",
      "result_data": [
        {
          "品类": "手机",
          "销量": 24,
          "销售额": 182000
        }
      ],
      "status": "success",
      "created_at": "2026-07-03T06:14:00"
    }
  ]
}
```

前端恢复为：

1. 用户消息：显示 `query`。
2. 助手消息：
   - `content` 显示摘要。
   - `result` 使用 `result_data`。
   - `MessageBubble` 继续复用当前结果表格组件。

也就是说，历史恢复后的展示效果要和刚查完时基本一致。

## 四、功能二：历史会话删除

### 4.1 当前问题

左侧历史会话只能查看，不能删除。

这会导致：

1. 错误请求留下的空会话无法清理。
2. 测试过程中产生大量重复会话。
3. 用户没有历史管理能力。

### 4.2 升级方案

新增后端接口：

```http
DELETE /api/sessions/{session_id}
```

删除逻辑：

1. 删除 `query_history` 中该 `session_id` 的所有记录。
2. 删除 `chat_session` 中该 `session_id` 的会话记录。
3. 返回删除结果。

返回示例：

```json
{
  "deleted": true,
  "session_id": "abc123"
}
```

### 4.3 后端 Repository 方法

在 `QueryHistoryRepository` 中新增：

```python
async def delete_session(self, session_id: str) -> None:
    """删除一个会话及其全部历史记录。"""
    await self.session.execute(
        text("DELETE FROM query_history WHERE session_id = :session_id"),
        {"session_id": session_id},
    )
    await self.session.execute(
        text("DELETE FROM chat_session WHERE session_id = :session_id"),
        {"session_id": session_id},
    )
    await self.session.commit()
```

### 4.4 后端 Service 方法

在 `QueryService` 中新增：

```python
async def delete_session(self, session_id: str) -> dict:
    await self.query_history_repository.ensure_tables()
    await self.query_history_repository.delete_session(session_id)
    return {"deleted": True, "session_id": session_id}
```

### 4.5 后端 Router 方法

在 `query_router.py` 中新增：

```python
@query_router.delete("/api/sessions/{session_id}")
async def delete_session_handler(
    session_id: str,
    query_service: Annotated[QueryService, Depends(get_query_service)],
):
    return await query_service.delete_session(session_id)
```

### 4.6 前端交互

左侧历史会话每一项增加一个删除按钮。

建议 UI：

```text
历史会话
┌─────────────────────────────┐
│ 统计 2025 年第一季度 GMV   × │
└─────────────────────────────┘
```

交互规则：

1. 点击会话主体：打开历史会话。
2. 点击删除按钮：不打开会话，只触发删除。
3. 删除前弹确认：
   ```text
   确认删除这个历史会话吗？
   ```
4. 用户确认后调用：
   ```http
   DELETE /api/sessions/{session_id}
   ```
5. 删除成功后刷新历史列表。
6. 如果删除的是当前正在查看的会话：
   - 清空当前消息。
   - 创建新的 `session_id`。
   - 输入框保持空。

### 4.7 空会话处理

对于之前异常产生的空会话：

1. 点击后如果没有 `history`，前端显示：
   ```text
   这个历史会话没有查询明细，可能是之前后端空流问题留下的记录。
   ```
2. 用户可以直接点击删除按钮清理。

## 七、测试方案

### 7.1 后端单元测试

测试点：

1. 查询成功时，`save_history()` 收到 `result_data`。
2. `get_session_history()` 返回 `result_data`。
3. 删除会话时，先删 `query_history`，再删 `chat_session`。
4. 删除不存在的会话不会报错。

### 7.2 前端手动测试

1. 启动系统。
2. 输入：
   ```text
   统计 2025 年 3 月各商品品类的销量和销售额
   ```
3. 查询成功后确认页面展示表格。
4. 点击“新会话”。
5. 在左侧历史会话中点击刚才的记录。
6. 预期：
   - 能恢复用户问题。
   - 能恢复表格结果。
   - 不再只显示“返回 6 行，字段...”。
7. 点击该历史会话的删除按钮。
8. 确认删除后：
   - 左侧历史会话消失。
   - 如果当前正在查看该会话，页面切换为空的新会话。

## 八、验收标准

完成后需要满足：

1. 新查询成功后，历史记录中保存 `result_data`。
2. 点击历史会话能恢复表格。
3. 历史会话可以删除。
4. 删除历史会话时，该会话下所有查询记录一并删除。
5. 删除当前会话后，前端不会停留在已删除会话。
6. 旧的空会话可以通过删除按钮清理。
7. 后端测试通过。
8. 前端类型检查通过。

## 九、边界说明

第一版不做：

1. 单独删除某一轮问答。
2. 批量删除全部历史。
3. 历史搜索。
4. 历史收藏。
5. 用户级权限隔离。

这些可以作为后续产品化能力继续升级。

# 问数 Agent 测评五阶段优化方案

## 1. 改造背景

当前项目已经具备自然语言问数、RAG 召回、SQL 生成与校验、多轮上下文、安全拒答和权限控制等能力，但最新真实测评结果表明，系统仍需要从“能够运行”进一步升级为“结果可信、边界明确、过程可测”。

最新测评基线如下：

| 指标 | 当前结果 |
| --- | ---: |
| 测评用例数 | 23 |
| 端到端成功率 | 52.17% |
| 字段 Recall@5 | 75.49% |
| 指标 Recall@5 | 70.59% |
| 字段值 Recall@5 | 100% |
| 表命中率 | 76.47% |
| 上下文通过率 | 70.59% |
| SQL 合规率 | 100% |
| SQL 可执行率 | 100% |
| 结果正确率 | 100% |
| 危险意图拦截率 | 100% |
| 无上下文拒答率 | 0% |
| 平均重试次数 | 0.76 |
| 平均延迟 / P95 延迟 | 0 ms / 0 ms，尚未正确采集 |

其中，端到端失败主要集中在三类问题：

1. **召回失败 7 条**：字段、指标或预期表未进入 TopK。
2. **领域边界失败 2 条**：数据库不存在主播、直播间、商圈等数据，系统仍尝试生成 SQL。
3. **行为判断失败 2 条**：业务澄清和变更方案的输出结构与测评预期不一致。

需要特别说明：当前 SQL 合规率、可执行率和结果正确率均为 100%，表示“已经成功生成 SQL 的适用用例表现稳定”，不代表全部 23 条用例都成功。端到端成功率 52.17% 才是当前整体能力的主要基线。

## 2. 改造目标

本轮优化不以修改测评规则来制造高分为目标，而是同时提升以下能力：

- 测评结果可以真实反映系统行为，不把合理的 SQL 写法误判为失败。
- 常见业务指标、字段和同义表达能够稳定召回。
- 面对数据库中不存在的数据主题时，系统能够明确拒答，不产生幻觉 SQL。
- 澄清、只读查询、变更方案和危险操作拒答具有稳定的结构化输出。
- 每次优化都能通过固定测评集与基线报告进行回归比较。

阶段性质量目标：

| 指标 | 目标 |
| --- | ---: |
| 端到端成功率 | >= 75% |
| 字段 Recall@5 | >= 85% |
| 指标 Recall@5 | >= 90% |
| 表命中率 | >= 90% |
| 无上下文拒答率 | >= 90% |
| 危险意图拦截率 | 100% |
| SQL 合规率 | >= 95% |
| SQL 可执行率 | >= 95% |

不要求追求 100% 端到端成功率。测评集规模有限，每条用例约影响 4.35 个百分点，应关注失败原因、长期趋势和关键能力是否退化。

## 3. 第一阶段：修正测评体系

### 3.1 目标

先保证测评工具本身可靠，区分“召回过程是否符合预期”和“最终业务答案是否正确”，避免因为实现路径不同而误判正确结果。

### 3.2 改造任务

1. 修复测评结果文件的 UTF-8 编码和 JSON 序列化问题，确保 `query_eval_result.json` 可以被标准 JSON 工具读取。
2. 在每条用例中记录总耗时、各节点耗时、模型调用次数和 SQL 重试次数，修复当前延迟指标始终为 0 的问题。
3. 重构端到端成功判定，将测评分为四层：
   - **行为层**：应该回答、澄清、拒答还是生成变更方案。
   - **召回层**：表、字段、指标、字段值是否命中。
   - **SQL 层**：SQL 是否只读、合规、可执行，并满足权限限制。
   - **结果层**：返回结果是否与标准结果在业务语义上等价。
4. 召回指标作为诊断指标保留，但不再要求所有正确 SQL 必须经过唯一固定的表和字段路径。
5. 支持一条用例配置多种合法 SQL 路径。例如直接使用事实表中的 `date_id` 完成时间过滤，与关联日期维表过滤都可以被视为合法实现。
6. 为每项指标明确分母：拒答类用例不参与 SQL 可执行率，回答类用例不参与危险操作拦截率。
7. 生成失败明细时，明确标记失败层级和原因，避免只输出一个总成功或失败状态。

### 3.3 验收标准

- 测评结果 JSON 能被 Python 和 PowerShell 正常解析。
- `avg_latency_ms` 和 `p95_latency_ms` 能反映真实耗时，不再固定为 0。
- SQL 结果正确但实现路径不同的用例，端到端结果可以通过，同时召回层仍如实记录未命中项。
- SQL 可执行但数值错误的用例必须判定为失败。
- 报告能够按行为、召回、SQL、结果四层统计失败数量。

## 4. 第二阶段：增强字段与指标召回

### 4.1 目标

提高常见电商业务表达的召回稳定性，减少模型因为缺少正确元数据而猜测字段、指标或表结构。

### 4.2 总体方案

采用“确定性业务词典 + Qdrant 向量召回 + 二次排序”的混合召回方式：

1. **确定性词典**处理明确的业务术语、缩写和同义词。
2. **Qdrant**负责处理自然语言近义表达和语义相似问题。
3. **二次排序**结合关键词命中、向量相似度、实体类型和业务关系选出最终 TopK。

### 4.3 改造任务

1. 建立并补充业务同义词映射，例如：

| 用户表达 | 标准业务含义 |
| --- | --- |
| 订单数、订单量 | `COUNT(order_id)` |
| 每天、每日、按日 | 日期 / `date_id` |
| 每月、月度、按月 | 月份字段 |
| 女性、男性 | `dim_customer.gender` |
| 黄金会员 | `dim_customer.member_level` |
| 品类、类目、商品类别 | `dim_product.category` |

2. 为指标知识补充名称、中文解释、英文缩写、计算公式、来源字段、适用维度和同义词。
3. 为字段元数据补充表名、字段名、中文含义、数据类型、示例值和关联关系。
4. 第一阶段适当扩大候选集，再通过二次排序压缩到最终 Top5，避免向量召回过早丢失正确候选。
5. 排序时提高精确关键词命中、标准指标名称命中和表关联可达性的权重。
6. 元数据发生变化后自动重建或增量更新 Qdrant 索引，并记录索引版本，避免 MySQL 元数据库与向量库内容不一致。
7. 为召回模块增加独立单元测试，不依赖完整 SQL 生成链路即可验证字段、指标和取值是否命中。

### 4.4 验收标准

- 字段 Recall@5 达到 85% 以上。
- 指标 Recall@5 达到 90% 以上。
- 表命中率达到 90% 以上。
- “订单数”“每日 GMV”“黄金会员订单数”“按品类统计销售额”等表达能够召回对应业务元数据。
- 元数据更新后，MySQL、Qdrant 和 Elasticsearch 中的版本状态可追踪。

## 5. 第三阶段：建立领域边界与无上下文拒答

### 5.1 目标

解决“数据库里没有相关数据，但模型仍根据相似表名生成 SQL”的问题。系统应先判断是否具备回答条件，再决定是否调用 SQL 生成模型。

### 5.2 改造任务

1. 从问题中提取核心业务概念，包括指标、分析对象、维度和过滤条件。
2. 对核心概念执行覆盖检查：只有核心概念能够被已授权的元数据、指标定义或字段值支持时，才能进入 SQL 生成节点。
3. 不能仅因为召回到 `fact_order` 等通用表就判定上下文充分。
4. 对以下数据库不存在的主题建立负样本：
   - 主播、直播间、直播转化率。
   - 线下门店、商圈、客流量。
   - 其他没有对应表、字段或指标定义的业务概念。
5. 上下文不足时返回结构化拒答，包括：
   - `status = unable_to_answer`。
   - `reason = no_recall_context`。
   - 缺失的数据主题或指标。
   - 可选的补充数据建议。
6. 明确区分三种情况：
   - 表达不清楚：进入业务澄清。
   - 数据域不存在：直接拒答。
   - 用户无权限：返回权限拒绝，不暴露未授权元数据。

### 5.3 验收标准

- 无上下文拒答率达到 90% 以上。
- “抖音直播间转化率最高的主播”等问题不再生成不存在的 `dim_anchor` 表。
- “线下门店客流量最高的商圈”等问题不再被错误映射到订单表。
- 拒答响应能够说明系统缺少哪类数据，但不泄露数据库内部敏感结构。

## 6. 第四阶段：完善行为判断与 SQL 链路

### 6.1 目标

让不同风险和不同意图的请求进入正确分支，同时降低无意义的大模型重试和查询延迟。

### 6.2 改造任务

1. 完善业务澄清输出，固定返回：
   - 澄清状态。
   - 澄清类型。
   - 缺失的业务槽位。
   - 可选择的澄清选项。
2. 对“哪些商品卖得最好”这类口径不明确的问题，要求用户选择按 GMV、销量还是订单数判断，不直接猜测指标。
3. 完善变更方案输出，固定返回：
   - 操作类型与目标对象。
   - 变更条件和拟修改内容。
   - 影响行数预估。
   - 风险等级与风险原因。
   - 审批状态、审批人和下一步动作。
4. 危险操作继续由后端确定性规则识别和拦截，不依赖大模型自行判断。
5. 生成 SQL 缺少 `LIMIT` 时，由后端 SQL 重写模块自动补充默认限制，不再调用大模型进行一次完整修正。
6. 只有语法错误、表字段错误和关联关系错误才进入大模型 SQL 修正节点。
7. SQL 执行前继续执行只读校验、权限校验、数据范围注入、`EXPLAIN` 校验和仓储层二次防护。
8. 记录每个节点的耗时、模型调用、SQL 修正原因和重试次数，用于定位性能瓶颈。

### 6.3 验收标准

- 模糊业务问题稳定进入澄清分支，并返回完整的结构化澄清内容。
- UPDATE、DELETE 等请求只展示方案或进入审批，不直接执行数据变更。
- TRUNCATE、DROP 等危险操作拦截率保持 100%。
- 缺少 `LIMIT` 不再触发大模型修正，平均重试次数明显下降。
- 所有执行 SQL 都通过权限校验和 SQL 安全校验。

## 7. 第五阶段：扩充真实业务测评集

### 7.1 目标

降低 23 条小样本带来的偶然性，使测评能够覆盖真实问数系统的主要业务路径和风险边界。

### 7.2 用例规模与分类

将测评集逐步扩充到 80 至 100 条，建议覆盖：

| 类型 | 重点验证内容 |
| --- | --- |
| 单指标查询 | GMV、订单数、AOV 等指标口径 |
| 多维分析 | 地区、商品、会员等级、时间等维度 |
| 趋势与对比 | 同比、环比、按日、按月、TopN |
| 条件过滤 | 时间、地区、类别、会员等级组合过滤 |
| 多轮追问 | 指标替换、地区替换、时间继承、指代消解 |
| 模糊问题 | 业务澄清与选项返回 |
| 未知领域 | 无上下文拒答和幻觉抑制 |
| 危险操作 | DROP、TRUNCATE、无条件 DELETE 等拦截 |
| 权限隔离 | 指标权限、字段权限和数据范围权限 |
| 空结果 | 空结果诊断与后续分析建议 |
| 大结果集 | LIMIT、分页、聚合和导出策略 |
| 同义改写 | 同一意图的多种自然语言表达 |

### 7.3 改造任务

1. 每条用例保存用户问题、会话上下文、用户权限、期望行为、期望指标、允许的表字段路径、结果断言和安全断言。
2. 为同一业务意图添加不同语言表达，验证系统不是只记住固定句式。
3. 固定一部分核心回归集，任何代码变更都必须运行。
4. 增加专项集合：召回集、上下文集、安全集、权限集和性能集。
5. 保存每次正式测评结果并与 baseline 对比，输出提升项、退化项和新增失败项。
6. 对依赖大模型的用例重复运行，观察结果波动，避免单次偶然成功被视为稳定能力。

### 7.4 验收标准

- 测评集达到 80 至 100 条，并覆盖上述主要场景。
- 核心回归集可以一条命令执行并生成报告。
- 报告同时展示总体指标、分类指标、失败原因和与 baseline 的变化。
- 安全、权限和未知领域的关键用例不得因总体分数提高而退化。

## 8. 实施顺序

五个阶段按以下顺序推进：

```text
修正测评口径
    -> 建立领域拒答
    -> 优化指标和字段召回
    -> 完善澄清、变更方案与 SQL 链路
    -> 扩充测评集
    -> 重新生成 baseline
```

先修正测评工具，是为了保证后续优化得到可信反馈；先修领域拒答，是为了避免继续生成幻觉 SQL；召回稳定后，再优化澄清、变更和性能，最后扩大样本验证系统的泛化能力。

## 9. 真实测评操作

### 9.1 保存当前基线

在 Windows CMD 中执行：

```bat
copy eval\reports\query_eval_result.json eval\reports\baseline_query_eval_result.json
```

### 9.2 运行真实测评

```bat
.\.venv\Scripts\python.exe -m eval.runners.run_eval --live
```

真实测评会调用大模型和实际依赖服务，运行前需要确认 MySQL、Qdrant、Elasticsearch、Embedding 服务和模型 API 均可用。

### 9.3 对比基线

若项目已提供报告对比命令，则执行：

```bat
.\.venv\Scripts\python.exe -m eval.runners.compare_reports --baseline eval\reports\baseline_query_eval_result.json --current eval\reports\query_eval_result.json --output eval\reports\query_eval_compare.md
```

如果当前尚未实现 `compare_reports`，应在第一阶段补充该工具，至少输出指标变化、修复用例、退化用例和新增失败原因。

## 10. 质量门禁

每次准备合并改造代码前，至少满足：

- 单元测试和核心回归集全部通过。
- 危险意图拦截率保持 100%。
- 不允许出现权限越权用例退化。
- SQL 合规率和可执行率不得低于 95%。
- 端到端成功率不能低于当前 baseline。
- 如果总体分数提升但关键安全用例失败，仍判定本次版本不通过。
- 报告 JSON 合法，失败明细和耗时数据完整。

## 11. 完成标准

当以下条件全部满足时，可认为五阶段改造完成：

1. 测评结果能够被稳定生成、解析和对比。
2. 端到端、召回、SQL、结果、安全和权限指标各自独立统计。
3. 常见电商字段和指标达到目标召回率。
4. 未知领域能够稳定拒答，不再生成幻觉表和字段。
5. 澄清、变更方案、拒答和查询结果均具有稳定的结构化响应。
6. SQL 默认限制和权限约束由后端确定性执行，大模型只负责候选生成和必要修正。
7. 测评集达到 80 至 100 条，并形成可重复执行的 baseline 回归流程。
8. 阶段性端到端成功率达到 75% 以上，同时安全和权限指标不退化。

完成后，项目的测评结论不再只是“生成的 SQL 能运行”，而是能够回答：系统是否理解了业务问题、是否召回了正确知识、是否在权限范围内生成了安全 SQL、是否返回了正确结果，以及面对无法回答的问题时是否能够可靠拒答。

## 12. 代码落地说明

本方案对应的代码改造已覆盖以下内容：

- 分层测评支持合法 SQL 替代路径、明确指标分母、失败层级统计和标准 UTF-8 JSON 输出。
- 真实 trace 记录总耗时、节点耗时、模型调用估算和 SQL 重试次数。
- baseline 对比报告增加已修复、新增退化和新增失败用例。
- 新增确定性电商业务词典，并与 Qdrant 向量召回、业务重排共同组成混合召回。
- 元数据全量构建使用统一索引版本，并重建 Qdrant 与 Elasticsearch 索引避免旧数据残留。
- SQL 生成前增加领域边界节点，未知领域不再进入 RAG 和 SQL 生成。
- SQL 缺少或超过限制时由后端补充或收紧 `LIMIT 1000`，安全策略错误不进入模型修正。
- 澄清事件增加缺失业务槽位，变更方案继续保持只展示、只预估、不执行。
- 测评集扩充到 80 条，并通过 `suite`、`tags`、`user_id` 和 `conversation` 描述专项场景。
- 增加质量门禁脚本，对总体阈值和安全、权限关键用例进行检查。

由于真实测评会产生模型费用并依赖本地服务，本次代码改造完成后应由项目使用者按第 9 节命令执行验证，再根据真实报告生成新的 baseline。

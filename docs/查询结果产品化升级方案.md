# 查询结果产品化升级方案

> **目标：** 让当前电商问数 Agent 不只是返回 SQL 表格，而是把查询结果升级为“可读、可分析、可视化、可导出”的数据分析产物。核心原则是：**规则计算事实，LLM 生成表达，系统校验结果**。

## 一、背景

当前系统已经具备：

1. 自然语言问数。
2. RAG 召回。
3. SQL 生成、校验、修正。
4. 查询结果表格展示。
5. 历史记录与结果回看。
6. 多轮上下文和 LLM 改写。

但查询结果目前主要还是表格，例如：

```text
返回 6 行，字段：商品品类、销量、销售额
```

这种结果对开发和验证 SQL 是够的，但对真实业务用户还不够产品化。业务用户更希望系统直接告诉他：

```text
销售额最高的是手机数码，销售额为 62190。
销量最高的是休闲零食，销量为 156。
手机数码销量不高但销售额最高，说明客单价明显更高。
```

因此，下一阶段应该升级查询结果展示能力。

## 二、设计原则

### 1. 必须体现 Agent 和大模型能力

本项目主打智能数据分析 Agent，因此结果摘要不应该只是简单规则模板。需要使用 LLM 生成更自然、更接近分析师表达的结果解读。

### 2. 大模型不能凭空分析

LLM 只能基于真实查询结果和规则计算出的事实进行表达，不能编造字段、数值、品类、趋势或结论。

### 3. 规则负责事实，LLM 负责表达

推荐分工：

```text
规则模块：
  - 行数
  - 字段
  - 最大值
  - 最小值
  - Top N
  - 趋势方向
  - 图表候选

LLM：
  - 生成自然语言摘要
  - 提炼洞察
  - 给出图表推荐理由
```

### 4. 输出必须结构化 JSON

LLM 不能直接返回自由文本，必须返回 JSON，方便前端展示和后端校验。

## 三、总体架构

```text
SQL 执行结果 result_data
  |
  v
规则事实计算 result_facts
  |
  v
LLM 生成分析摘要 result_analysis
  |
  v
JSON 解析与事实校验
  |
  v
前端展示
  - 智能摘要
  - 洞察列表
  - 自动图表
  - 表格
  - 导出按钮
```

## 四、结果事实计算

新增模块：

```text
app/core/result_productization/fact_extractor.py
```

职责：从 `result_data` 中提取确定性事实。

### 1. 输入示例

```json
[
  {"商品品类": "手机数码", "销量": 10, "销售额": 62190},
  {"商品品类": "食品饮料", "销量": 139, "销售额": 1115},
  {"商品品类": "休闲零食", "销量": 156, "销售额": 630}
]
```

### 2. 输出示例

```json
{
  "row_count": 6,
  "columns": ["商品品类", "销量", "销售额"],
  "dimension_columns": ["商品品类"],
  "metric_columns": ["销量", "销售额"],
  "top_values": {
    "销量": {
      "label": "休闲零食",
      "value": 156
    },
    "销售额": {
      "label": "手机数码",
      "value": 62190
    }
  },
  "chart_candidates": [
    {
      "type": "bar",
      "x": "商品品类",
      "y": "销售额",
      "reason": "分类维度 + 数值指标，适合柱状图对比"
    }
  ]
}
```

### 3. 字段判断规则

第一版可以用轻量规则：

| 字段类型 | 判断方式 |
| --- | --- |
| 维度字段 | 字符串类型，或字段名包含品类、商品、大区、会员、日期 |
| 指标字段 | 数字类型，或字段名包含 GMV、销售额、销量、订单数 |
| 时间字段 | 字段名包含日期、时间，或值形如 `20250301` |

## 五、LLM 分析摘要

新增模块：

```text
app/core/result_productization/analysis_writer.py
```

职责：把 `result_facts` 和少量 `result_data` 交给 LLM，生成结构化分析摘要。

### 1. LLM 输入

```json
{
  "query": "统计 2025 年 3 月各商品品类的销量和销售额",
  "resolved_query": "统计 2025 年 3 月各商品品类的销量和销售额",
  "result_facts": {
    "row_count": 6,
    "columns": ["商品品类", "销量", "销售额"],
    "top_values": {
      "销量": {"label": "休闲零食", "value": 156},
      "销售额": {"label": "手机数码", "value": 62190}
    }
  },
  "sample_rows": [
    {"商品品类": "手机数码", "销量": 10, "销售额": 62190},
    {"商品品类": "食品饮料", "销量": 139, "销售额": 1115}
  ]
}
```

### 2. LLM 输出

LLM 必须输出 JSON：

```json
{
  "summary": "本次查询统计了 2025 年 3 月各商品品类的销量和销售额，共返回 6 个品类。",
  "insights": [
    "销售额最高的是手机数码，销售额为 62190。",
    "销量最高的是休闲零食，销量为 156。",
    "手机数码销量不高但销售额最高，说明该品类客单价明显更高。"
  ],
  "chart_recommendation": {
    "type": "bar",
    "x": "商品品类",
    "y": "销售额",
    "reason": "商品品类是分类维度，销售额是数值指标，适合用柱状图对比。"
  }
}
```

### 3. Prompt 约束

Prompt 必须强调：

```text
你是电商数据分析 Agent 的结果解读模块。

要求：
1. 只能基于给定 result_facts 和 sample_rows 生成结论。
2. 不能编造不存在的字段、品类、数值。
3. 所有数字必须来自 result_facts 或 sample_rows。
4. 如果数据不足以支持某个结论，不能强行分析。
5. 必须输出 JSON，不能输出 Markdown。
6. insights 最多 3 条，保持简洁。
```

## 六、事实校验

新增模块：

```text
app/core/result_productization/analysis_validator.py
```

职责：校验 LLM 分析结果是否越界。

### 1. 校验内容

| 校验项 | 说明 |
| --- | --- |
| JSON 格式 | 必须是合法 JSON |
| 字段校验 | 不允许出现 result_data 中不存在的字段 |
| 数值校验 | 关键数字必须能在 `result_facts` 中找到 |
| 图表字段校验 | `x` 和 `y` 必须属于结果字段 |
| 洞察数量 | insights 不超过 3 条 |

### 2. 校验失败兜底

如果 LLM 输出不合法，不能让整个查询失败。

推荐兜底：

```json
{
  "summary": "本次查询返回 6 行结果，字段包括：商品品类、销量、销售额。",
  "insights": [
    "销售额最高的是手机数码，销售额为 62190。",
    "销量最高的是休闲零食，销量为 156。"
  ],
  "chart_recommendation": {
    "type": "bar",
    "x": "商品品类",
    "y": "销售额",
    "reason": "分类维度和数值指标适合柱状图。"
  },
  "generated_by": "rule_fallback"
}
```

## 七、后端接口与 SSE 事件

查询执行成功后，除了返回 `result` 事件，还可以新增 `result_analysis` 事件。

### 1. SSE 事件示例

```json
{
  "type": "result_analysis",
  "data": {
    "summary": "本次查询统计了 2025 年 3 月各商品品类的销量和销售额，共返回 6 个品类。",
    "insights": [
      "销售额最高的是手机数码，销售额为 62190。",
      "销量最高的是休闲零食，销量为 156。"
    ],
    "chart_recommendation": {
      "type": "bar",
      "x": "商品品类",
      "y": "销售额",
      "reason": "商品品类是分类维度，销售额是数值指标，适合柱状图对比。"
    }
  }
}
```

### 2. 推荐发送顺序

```text
progress
result
result_analysis
```

也可以先返回表格，再异步生成分析摘要。第一版建议同步生成，便于实现和测试。

## 八、历史记录扩展

建议在 `query_history` 表增加字段：

```sql
result_facts JSON NULL COMMENT '查询结果事实摘要',
result_analysis JSON NULL COMMENT 'LLM 生成的结果分析摘要'
```

历史回看时，前端可以恢复：

1. 表格。
2. 智能摘要。
3. 洞察列表。
4. 图表推荐。

## 九、前端展示方案

新增组件：

```text
frontend/src/components/ResultAnalysisCard.tsx
frontend/src/components/ResultChart.tsx
```

### 1. 展示结构

```text
查询完成，共 6 行结果。

智能摘要
本次查询统计了 2025 年 3 月各商品品类的销量和销售额，共返回 6 个品类。

关键洞察
- 销售额最高的是手机数码，销售额为 62190。
- 销量最高的是休闲零食，销量为 156。
- 手机数码销量不高但销售额最高，说明客单价明显更高。

图表
[柱状图：商品品类 vs 销售额]

表格
商品品类 | 销量 | 销售额

操作
[复制] [导出 CSV]
```

### 2. 图表类型判断

第一版支持：

| 数据形态 | 图表类型 |
| --- | --- |
| 时间字段 + 数值字段 | 折线图 |
| 分类字段 + 数值字段 | 柱状图 |
| 分类字段 + 占比/金额字段 | 饼图可选，但默认柱状图 |

推荐优先使用柱状图和折线图，少用饼图。

### 3. 图表库

推荐使用：

```text
recharts
```

原因：

1. React 生态成熟。
2. 接入简单。
3. 足够满足柱状图、折线图需求。

## 十、CSV 导出

第一版导出可以纯前端实现，不需要新增后端接口。

实现位置：

```text
frontend/src/lib/exportCsv.ts
```

功能：

1. 接收 `message.result`。
2. 自动读取字段。
3. 生成 CSV。
4. 创建下载链接。

按钮位置：

```text
ResultTable 或 MessageBubble 操作区
```

按钮：

```text
导出 CSV
```

## 十三、实施顺序

建议分四步：

### 第一阶段：事实计算

1. 新增 `fact_extractor.py`。
2. 根据 `result_data` 计算 `result_facts`。
3. 写单元测试。

### 第二阶段：LLM 分析摘要

1. 新增 `analysis_writer.py`。
2. 调用 LLM 生成 JSON 摘要。
3. 新增 `analysis_validator.py`。
4. 校验失败时规则兜底。

### 第三阶段：接入 QueryService

1. SQL 执行结果返回后生成 `result_analysis`。
2. SSE 返回 `result_analysis`。
3. 历史表保存 `result_facts` 和 `result_analysis`。

### 第四阶段：前端产品化展示

1. 新增 `ResultAnalysisCard`。
2. 新增 `ResultChart`。
3. 新增 CSV 导出。
4. 历史回看恢复分析摘要和图表。

## 十四、验收标准

完成后需要满足：

1. 查询结果不只展示表格，还展示智能摘要。
2. 摘要由 LLM 生成，但只能基于真实结果事实。
3. 关键数字必须来自 `result_data` 或 `result_facts`。
4. LLM 失败时不会影响表格展示。
5. 至少支持柱状图和折线图。
6. 支持 CSV 导出。
7. 历史回看能恢复摘要、图表和表格。
8. 后端测试覆盖事实计算、LLM 校验和兜底。

```text
智能摘要

关键洞察

2. 销量最高的是休闲零食，销量为 156。
3. 手机数码销量不高但销售额最高，说明该品类客单价明显更高。

图表
柱状图：商品品类 vs 销售额

表格

```

这时系统就不再只是“SQL 查询结果展示器”，而更像一个能解释结果、推荐图表、支持导出的智能数据分析 Agent。

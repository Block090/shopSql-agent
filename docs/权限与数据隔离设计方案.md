# 权限与数据隔离设计方案

## 1. 背景

当前项目已经实现自然语言问数 Agent 的核心链路，包括意图识别、RAG 元数据召回、SQL 生成、SQL 校验、SQL 修正、查询执行和测评闭环。

但在企业真实场景中，仅能生成正确 SQL 还不够，还需要解决以下问题：

- 不同用户能访问的数据范围不同，例如区域运营只能看自己负责的大区。
- 部分字段属于敏感字段，例如手机号、姓名、地址等，不应该直接暴露。
- 部分指标可能只允许特定角色查看，例如利润、成本、退款率等。
- 权限判断不能依赖大模型，因为大模型可能幻觉、遗漏条件或错误理解用户权限。

因此需要新增一个后端确定性的权限与数据隔离模块。

## 2. 设计目标

本方案的核心目标是：

> 大模型只负责理解问题和生成候选 SQL，最终能不能查、能查哪些数据，必须由后端权限系统和 SQL 校验逻辑决定。

具体目标：

- 支持根据用户身份生成权限上下文。
- 支持基于角色限制可访问表、字段、指标和数据范围。
- 在 RAG 召回后过滤无权限元数据，避免敏感字段进入 Prompt。
- 在 SQL 执行前做权限校验，避免越权 SQL 被执行。
- 对高风险、越权或权限不明确的查询进行拒答、澄清或转人工。

## 3. 核心原则

### 3.1 权限不交给大模型判断

大模型可以参与：

- 理解用户问题。
- 生成候选 SQL。
- 辅助生成解释说明。

大模型不能负责：

- 判断用户是否有权限。
- 决定用户能访问哪些字段。
- 决定是否允许执行 SQL。
- 自行追加权限条件后直接放行。

最终授权必须由后端确定性规则完成。

### 3.2 敏感元数据不进入 Prompt

如果用户没有某个字段或指标的访问权限，那么该字段或指标不应该出现在 SQL 生成上下文中。

例如用户没有手机号权限，即使 RAG 召回到 `customer_phone`，也要在进入 `generate_sql` 节点前过滤掉。

这样可以降低模型生成越权 SQL 的概率。

### 3.3 SQL 执行前必须二次校验

即使前面已经过滤过上下文，也不能完全相信模型输出。

SQL 执行前必须检查：

- 是否访问无权限表。
- 是否访问敏感字段。
- 是否使用无权限指标。
- 是否缺少用户数据范围过滤条件。
- 是否绕过数据隔离规则。

只有通过权限校验和 SQL 安全校验后，才能执行查询。

## 4. 整体流程

升级后的流程如下：

```text
用户请求
  -> FastAPI 接收请求
  -> 根据 user_id 加载权限上下文
  -> LangGraph 进入问数流程
  -> 意图识别
  -> RAG 召回字段、指标、字段值
  -> 权限过滤 RAG 召回结果
  -> 二次过滤表和指标
  -> 大模型生成候选 SQL
  -> SQL Guard 基础安全校验
  -> SQL 权限校验 / 数据范围注入
  -> MySQL EXPLAIN 校验
  -> 执行 SQL
  -> 返回结果和查询依据
```

核心新增两道关：

```text
RAG 后权限过滤
SQL 执行前权限校验
```

## 5. 权限上下文设计

新增 `permission_context`，用于描述当前用户的数据访问权限。

示例：

```json
{
  "user_id": "u_001",
  "role": "region_operator",
  "allowed_regions": ["华东"],
  "allowed_tables": ["fact_order", "dim_region", "dim_date", "dim_product"],
  "allowed_metrics": ["GMV", "订单数", "AOV"],
  "denied_columns": ["customer_phone", "customer_name", "customer_address"],
  "data_scope": {
    "region_name": ["华东"]
  },
  "risk_level": "normal"
}
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `user_id` | 当前用户 ID |
| `role` | 当前用户角色 |
| `allowed_regions` | 可访问大区 |
| `allowed_tables` | 可访问表 |
| `allowed_metrics` | 可访问指标 |
| `denied_columns` | 禁止访问的敏感字段 |
| `data_scope` | 强制数据范围过滤条件 |
| `risk_level` | 默认风险等级 |

当前项目可以先通过配置文件模拟权限，后续真实落地时可以接入 RBAC 权限表、组织架构系统或企业统一认证系统。

## 6. 三层权限控制

### 6.1 请求层权限判断

请求进入 FastAPI 后，先获取用户身份。

可以通过请求头传入：

```text
X-User-Id: u_001
```

后端根据 `user_id` 加载权限上下文。

如果没有用户身份，直接拒绝查询：

```text
当前请求缺少用户身份，无法进行数据查询。
```

如果用户请求明显涉及敏感字段或越权数据，例如：

```text
导出所有用户手机号
```

可以在进入 SQL 生成前直接拒绝。

这一层解决：

- 不能匿名查询数据。
- 不能让模型决定用户身份。
- 明显敏感请求不进入后续链路。

### 6.2 RAG 召回后权限过滤

RAG 召回只能说明“这些元数据和问题相关”，不能说明“用户有权限使用这些元数据”。

因此 RAG 召回后需要进行权限过滤。

过滤对象包括：

- 表信息 `table_infos`
- 字段信息 `columns`
- 指标信息 `metric_infos`
- 字段值信息 `retrieved_value_infos`
- 相似查询案例 `similar_query_cases`

过滤规则示例：

- 不在 `allowed_tables` 中的表直接移除。
- 在 `denied_columns` 中的字段直接移除。
- 不在 `allowed_metrics` 中的指标直接移除。
- 涉及敏感字段的 Query Case 不作为 SQL 生成参考。

示例：

```text
RAG 召回：
dim_customer.customer_phone

权限配置：
denied_columns = ["customer_phone"]

处理结果：
customer_phone 不进入 generate_sql 上下文
```

这一层的价值是：

> 模型在生成 SQL 时只能看到当前用户被授权的表、字段和指标。

### 6.3 SQL 执行前权限校验

SQL 生成后，需要在执行前进行确定性权限校验。

检查内容：

- SQL 是否访问了无权限表。
- SQL 是否选择了敏感字段。
- SQL 是否使用了无权限指标的依赖字段。
- SQL 是否缺少数据范围条件。
- SQL 是否查询了超出用户权限的数据范围。

例如用户只有华东权限，但模型生成：

```sql
SELECT SUM(order_amount) AS GMV
FROM fact_order
LIMIT 100;
```

该 SQL 缺少地区范围限制，不能直接执行。

处理方式有两种：

1. 拒绝执行，提示权限条件缺失。
2. 在确定安全的情况下，由后端强制追加权限条件。

推荐策略：

- 查询范围不明确时拒绝或澄清。
- 权限范围明确且可安全注入时，由后端追加条件。
- 不让大模型自行决定是否追加权限条件。

示例：

```sql
WHERE region_name = '华东'
```

或：

```sql
WHERE region_id IN (...)
```

## 7. 风险分级策略

权限模块可以和已有查询风险分级能力结合。

### 7.1 低风险

特征：

- 普通聚合查询。
- 只读 SELECT。
- 不涉及敏感字段。
- 不越权。
- 命中已配置指标口径。
- SQL 通过安全校验和权限校验。

处理方式：

```text
自动执行
```

### 7.2 中风险

特征：

- 查询范围较大。
- 业务口径不够明确。
- 结果可能影响业务决策。
- 权限范围需要进一步确认。

处理方式：

```text
要求澄清 / 展示风险提示 / 展示查询依据
```

### 7.3 高风险

特征：

- 涉及敏感字段。
- 涉及数据导出。
- 涉及增删改。
- 涉及越权访问。
- 缺少必要权限条件。

处理方式：

```text
拒答 / 转人工 / 生成变更方案但不执行
```

## 8. 与现有模块的结合点

### 8.1 FastAPI 接口层

在请求进入时获取用户身份。

可扩展：

```json
{
  "query": "统计 2025 年第一季度 GMV",
  "session_id": "xxx",
  "user_id": "u_001"
}
```

或从请求头获取：

```text
X-User-Id: u_001
```

### 8.2 DataAgentState

新增状态字段：

```python
permission_context
permission_error
permission_risk_level
```

用于在 LangGraph 节点间传递权限信息和权限校验结果。

### 8.3 LangGraph 流程

建议新增节点：

```text
load_permission_context
filter_permission_context
validate_sql_permission
reject_permission_denied
```

推荐流程：

```text
check_query_intent
-> load_permission_context
-> extract_keywords
-> recall_column / recall_metric / recall_value
-> merge_retrieved_info
-> filter_permission_context
-> filter_table / filter_metric
-> generate_sql
-> validate_sql
-> validate_sql_permission
-> run_sql
```

如果权限校验失败：

```text
validate_sql_permission -> reject_permission_denied
```

### 8.4 RAG 模块

在 `merge_retrieved_info` 后，对候选上下文做权限裁剪。

权限裁剪后再进入：

```text
filter_table
filter_metric
generate_sql
```

这样可以避免无权限字段进入 Prompt。

### 8.5 SQL 校验模块

在现有 SQL Guard 和 EXPLAIN 校验基础上，新增 SQL 权限校验。

现有校验：

```text
只允许 SELECT
禁止危险关键字
必须 LIMIT
EXPLAIN 检查语法和表字段
```

新增校验：

```text
表权限
字段权限
指标权限
数据范围权限
敏感字段拦截
```

## 9. 示例场景

### 9.1 普通低风险查询

用户权限：

```json
{
  "allowed_regions": ["华东"],
  "allowed_metrics": ["GMV", "订单数"]
}
```

用户问题：

```text
统计华东地区 2025 年第一季度 GMV
```

处理结果：

```text
命中授权地区
命中授权指标
SQL 通过校验
自动执行
```

### 9.2 查询范围越权

用户权限：

```json
{
  "allowed_regions": ["华东"]
}
```

用户问题：

```text
统计全国 2025 年第一季度 GMV
```

处理结果：

```text
用户仅有华东权限
全国查询越权
拒绝执行或提示只能查询华东范围
```

推荐返回：

```text
当前账号仅有华东地区数据权限，无法查询全国数据。
```

### 9.3 敏感字段查询

用户问题：

```text
导出所有用户手机号
```

处理结果：

```text
命中敏感字段 customer_phone
高风险
拒绝执行
```

推荐返回：

```text
该请求涉及敏感字段，当前权限不允许直接查询。
```

### 9.4 模型遗漏权限条件

用户权限：

```json
{
  "allowed_regions": ["华东"]
}
```

模型生成 SQL：

```sql
SELECT SUM(order_amount) AS GMV
FROM fact_order
LIMIT 100;
```

权限校验结果：

```text
SQL 缺少 region_name = '华东' 或 region_id 范围条件
不允许直接执行
```

处理方式：

```text
拒绝执行或由后端安全注入权限条件
```

## 10. 测评设计

新增权限相关测评用例。

### 10.1 用例类型

- 授权范围内查询。
- 越权大区查询。
- 敏感字段查询。
- 无用户身份查询。
- 权限条件缺失 SQL。
- 变更类高风险操作。

### 10.2 关键指标

新增测评指标：

| 指标 | 说明 |
| --- | --- |
| `permission_block_rate` | 越权请求拦截率 |
| `sensitive_field_block_rate` | 敏感字段拦截率 |
| `data_scope_enforcement_rate` | 数据范围强制命中率 |
| `authorized_query_success_rate` | 授权查询成功率 |
| `permission_false_reject_rate` | 合法请求误拒率 |

### 10.3 测评目标

权限测评的目标不是让所有请求都成功，而是：

- 合法查询可以正常执行。
- 越权查询必须被拦截。
- 敏感字段不能进入 SQL 执行。
- 权限范围不能依赖大模型自觉遵守。

## 11. 面试表达

可以这样回答权限相关问题：

> 针对权限问题，我会把权限控制做成后端确定性模块，而不是依赖大模型判断。请求进入后先根据用户身份生成权限上下文，包括可访问指标、表字段、敏感字段和数据范围。RAG 召回后会先过滤掉用户无权限的元数据，避免敏感字段进入模型上下文。SQL 生成后，还会在执行前做权限校验，检查是否访问无权限字段、是否缺少数据范围条件，必要时由后端强制追加权限过滤或直接拒绝执行。这样大模型只负责生成候选 SQL，最终授权由后端规则系统控制。

简历表述：

> 设计权限与数据隔离方案，将用户角色、可访问指标、敏感字段和数据范围纳入权限上下文，在 RAG 召回后和 SQL 执行前进行双层权限校验，避免依赖大模型进行权限判断。

## 12. 后续实现优先级

### P0：最小可用权限控制

- 新增用户权限配置文件。
- 请求中携带 `user_id`。
- 加载 `permission_context`。
- RAG 后过滤无权限字段和指标。
- SQL 执行前拦截敏感字段。

### P1：数据范围隔离

- 支持按大区、门店、类目等维度限制数据范围。
- SQL 执行前检查是否包含必要权限条件。
- 明确场景下支持后端注入权限过滤条件。

### P2：权限测评闭环

- 新增权限测评集。
- 统计越权拦截率、敏感字段拦截率和合法查询成功率。
- 将权限测评结果纳入 baseline 对比。

### P3：企业级扩展

- 接入真实登录态。
- 接入 RBAC 权限表。
- 接入组织架构和数据权限系统。
- 对接审批系统处理高风险请求。

# 真实业务场景增强设计方案

## 1. 背景

当前项目已经具备自然语言问数、RAG 召回、SQL 生成校验、多轮上下文、历史会话、结果分析和前端展示等能力。

但是如果要把项目作为简历项目或面试项目来讲，仅仅强调“能用自然语言生成 SQL”还不够。真实企业里的数据分析系统不仅要能查数，还要解决这些问题：

- 结果是否可信。
- 指标口径是否明确。
- 数据是否安全。
- 查询为空时如何解释。
- 用户看到结果后下一步该怎么分析。
- 普通业务用户是否会误触敏感数据。

因此，后续增强不应该为了功能数量而堆功能，而应该围绕真实电商经营分析场景，补齐可信、安全、可解释和分析引导能力。

## 2. 设计原则

### 2.1 不为加功能而加功能

每个新增能力都必须回答一个问题：

> 这个功能解决了哪个真实业务问题？

如果只是“看起来更酷”，就不优先做。

如果能解决以下问题，就值得做：

- 减少指标口径误解。
- 提升查询结果可信度。
- 保护敏感数据。
- 帮助用户理解空结果。
- 引导用户继续分析。
- 降低业务用户使用门槛。

### 2.2 面向业务用户，而不是只面向开发者

普通业务用户不一定关心 SQL、表名和字段名，他们更关心：

- 这个指标是什么意思。
- 这个结果怎么算出来。
- 本次分析用了什么时间范围。
- 本次结果是否有异常。
- 接下来应该看哪个维度。

因此，系统解释信息应优先使用业务语言，而不是直接暴露数据库内部结构。

### 2.3 分层解释，控制数据库暴露风险

查询解释不能简单等于展示完整 SQL 和底层表字段。

更合理的方式是分层展示：

- 业务用户看到指标口径、分析维度、时间范围、过滤条件。
- 管理员或数据分析师看到脱敏 SQL、命中字段、召回详情、执行耗时。

这样既提升可信度，又避免暴露内部数据库结构和敏感字段。

## 3. 推荐增强方向

## 3.1 分层查询依据说明

### 业务问题

用户看到结果后，常常会问：

- 这个结果是怎么算出来的？
- 用的是哪个指标口径？
- 本次筛选了哪些条件？
- 结果是否可信？

但如果直接展示完整 SQL 和底层字段，又可能暴露数据库结构和敏感信息。

### 设计方案

将查询解释分为两层。

业务用户看到：

- 指标口径。
- 分析维度。
- 时间范围。
- 过滤条件。
- 结果摘要。

管理员或数据分析师看到：

- 脱敏后的 SQL。
- 命中的表和字段。
- 召回命中详情。
- 执行耗时。
- 上下文改写结果。

### 示例

用户问题：

```text
统计 2025 年第一季度各大区 GMV
```

业务解释：

```text
指标：GMV
口径：已支付订单金额汇总
分析维度：大区
时间范围：2025-01-01 至 2025-03-31
过滤条件：已支付订单
```

技术解释：

```text
生成 SQL：仅管理员可见，且敏感字段脱敏
命中字段：订单日期、大区、支付金额
召回来源：字段召回、指标召回、字段取值召回
```

### 简历表达

```text
设计分层查询依据说明，面向业务用户展示指标口径、分析维度和过滤条件，面向管理员展示脱敏 SQL 和召回详情，在提升结果可信度的同时控制数据库结构暴露风险。
```

## 3.2 指标口径确认机制

### 业务问题

电商分析中很多指标存在口径差异。

例如：

- GMV：下单或成交金额。
- 支付销售额：实际支付金额。
- 净销售额：支付金额扣除退款。
- 订单数：全部订单或支付订单。

如果用户只说“销售额”，系统直接猜一个口径，可能导致结果错误。

### 设计方案

当用户问题命中易歧义指标时，系统不直接生成 SQL，而是先触发口径澄清。

例如用户问：

```text
统计本月销售额
```

系统追问：

```text
你想查看哪种销售额口径？
1. GMV：成交总额
2. 支付销售额：实际支付金额
3. 净销售额：支付金额扣除退款
```

用户确认后，再进入后续 RAG 召回和 SQL 生成。

### 简历表达

```text
实现指标口径确认机制，对销售额、GMV、订单数等易歧义指标进行澄清，避免自然语言问数中的指标口径误判。
```

## 3.3 敏感字段识别和脱敏

### 业务问题

电商数据中包含大量敏感信息：

- 手机号。
- 收货地址。
- 用户 ID。
- 订单号。
- 支付流水号。
- 成本价。
- 利润。

这些信息不应该随意展示、导出或暴露在查询解释中。

### 设计方案

在元数据中为字段增加敏感级别或字段标签。

示例：

```text
phone：敏感，手机号
address：敏感，地址
user_id：中敏，用户标识
cost_price：高敏，成本价
profit：高敏，利润
```

系统根据敏感级别采取不同策略：

- 普通展示时脱敏。
- 查询解释中使用业务别名。
- 高敏字段默认不展示。
- 导出时再次检查。
- 高风险查询触发提示或拒答。

示例脱敏：

```text
13812345678 -> 138****5678
详细地址 -> 地址已隐藏
user_id -> 用户标识
```

### 简历表达

```text
设计敏感字段识别与结果脱敏机制，对手机号、地址、用户标识等敏感数据进行展示和导出控制，提升问数系统的数据安全性。
```

## 3.4 查询风险分级

### 业务问题

不同查询风险不同。

例如：

- 汇总 GMV：低风险。
- 查询用户明细：中风险。
- 查询手机号或地址：高风险。
- 导出大量明细：高风险。
- 数据变更：极高风险。

系统需要根据风险等级采取不同策略，而不是所有请求都一视同仁。

### 设计方案

根据以下因素评估风险：

- 查询是否为汇总还是明细。
- 是否命中敏感字段。
- 预计返回行数。
- 是否包含导出行为。
- 是否为数据变更意图。

风险策略：

| 风险等级 | 示例 | 处理策略 |
| --- | --- | --- |
| 低风险 | 查询各大区 GMV | 直接返回 |
| 中风险 | 查询订单明细 | 增加提示或限制行数 |
| 高风险 | 查询手机号、地址 | 脱敏、拒答或需要权限 |
| 极高风险 | UPDATE、DELETE | 进入审批方案，不直接执行 |

### 简历表达

```text
设计查询风险分级机制，根据查询粒度、字段敏感性、结果行数和操作类型区分低/中/高风险，并采取提示、脱敏、拒答或审批策略。
```

## 3.5 空结果诊断

### 业务问题

用户查询后返回 0 行时，如果系统只说“没有数据”，体验很差。

空结果可能由多种原因导致：

- 时间范围无数据。
- 过滤条件过窄。
- 字段取值不匹配。
- 指标口径不适用。
- 数据同步尚未完成。

### 设计方案

当 SQL 查询结果为空时，系统生成空结果诊断。

诊断方向：

- 时间条件是否过窄。
- 维度条件是否匹配真实字段取值。
- 是否可以放宽过滤条件。
- 是否可以查询相邻时间段。
- 是否可能是数据未同步。

示例：

```text
本次查询返回 0 行，可能原因：
1. 2026 年 12 月暂无销售数据。
2. “华东一区”可能不是数据库中的标准地区取值。
3. 当前过滤条件同时限定了地区和品类，范围较窄。

你可以尝试：
- 改查 2025 年同期。
- 放宽地区条件为“华东”。
- 先查看该时间段是否存在订单数据。
```

### 简历表达

```text
实现空结果诊断能力，对返回 0 行的查询从时间范围、过滤条件、字段取值和指标适用性等角度生成原因分析和改写建议。
```

## 3.6 结果异常提示

### 业务问题

业务用户不只是想知道“是多少”，还想知道：

- 有没有异常。
- 哪里明显偏高或偏低。
- 是否需要进一步排查。

例如：

- GMV 环比下降 30%。
- 某地区订单数为 0。
- 某品类销售额异常高。
- 某个会员等级贡献占比异常。

### 设计方案

在结果分析阶段增加异常检测规则。

可先从简单规则开始：

- 最大值和最小值差异过大。
- 某维度为 0。
- TOP 1 占比过高。
- 查询结果为空。
- 同比或环比变化超过阈值。

示例：

```text
本次结果中，华南地区 GMV 明显低于其他区域，建议继续按品类拆分分析，排查是否为某些品类销售下降导致。
```

### 简历表达

```text
实现查询结果异常提示，基于结果分布、极值、空值和同比/环比变化识别潜在经营异常，并生成后续分析建议。
```

## 3.7 后续分析建议

### 业务问题

很多业务用户看到结果后，不知道下一步应该怎么分析。

例如用户查询：

```text
统计各大区 GMV
```

得到结果后，下一步可能需要分析：

- 哪个地区下降。
- 下降是订单数减少还是客单价下降。
- 哪些品类影响最大。
- 是否和会员结构有关。

### 设计方案

基于用户问题和查询结果生成后续分析建议。

示例：

```text
你可以继续分析：
1. 按品类拆分华东地区 GMV。
2. 对比各地区订单数和客单价。
3. 查看 GMV 下降地区的 TOP 商品变化。
4. 按会员等级分析高价值用户贡献。
```

### 简历表达

```text
基于查询结果生成后续分析建议，引导用户从地区、品类、会员、订单数、客单价等维度继续拆解经营问题。
```

## 3.8 指标血缘和依赖说明

### 业务问题

当用户看到一个指标时，可能会追问：

- 这个指标从哪里来？
- 依赖哪些字段？
- 计算逻辑是什么？
- 适用哪些过滤条件？

### 设计方案

为指标提供业务血缘和技术血缘两类说明。

业务用户看到：

```text
GMV 来源于已支付订单金额汇总。
```

技术用户看到：

```text
GMV -> fact_order.pay_amount
时间过滤 -> fact_order.order_date
区域聚合 -> dim_region.region_name
```

### 简历表达

```text
设计指标血缘说明能力，展示指标依赖字段、计算逻辑和适用过滤条件，支持业务口径复核和数据问题追踪。
```

## 3.9 查询模板沉淀

### 业务问题

业务用户经常重复问一些固定类型的问题。

例如：

- 经营日报。
- 区域销售分析。
- 商品 TOP 榜。
- 会员分层分析。
- 品类销售趋势。

如果每次都让用户从零开始写问题，使用门槛较高。

### 设计方案

沉淀常用分析模板。

模板不是固定报表，而是推荐问题、默认指标、默认维度的组合。

示例：

```text
经营概览：
- 统计本月 GMV、订单数、客单价。
- 对比本月和上月 GMV 变化。
- 查看近 7 天销售趋势。

商品分析：
- 查询销售额最高的前 10 个商品。
- 分析各品类销售额和销量。
- 查询低销量高库存商品。
```

### 简历表达

```text
沉淀电商常用分析模板，覆盖经营日报、区域销售、商品 TOP 榜和会员分层分析，降低业务用户自然语言提问成本。
```

## 3.10 数据新鲜度提示

### 业务问题

业务用户经常关心：

- 数据更新到什么时候？
- 当前结果是实时数据还是 T+1 数据？
- 今天的数据是否完整？

如果不展示数据新鲜度，用户可能误解结果。

### 设计方案

在查询结果和查询依据中展示数据更新时间。

示例：

```text
数据更新时间：2025-03-31 23:59
同步周期：T+1
说明：今日数据可能尚未完全入仓
```

### 简历表达

```text
增加数据新鲜度提示，展示查询所依赖数据的最近更新时间和同步周期，降低业务用户对数据时效的误解。
```

## 4. 推荐实施优先级

### 第一阶段：可信

目标：让用户知道结果怎么算出来。

- 分层查询依据说明。
- 指标口径展示。
- SQL 和字段脱敏。

### 第二阶段：安全

目标：避免敏感数据和高风险查询被随意暴露。

- 敏感字段识别。
- 查询风险分级。
- 导出控制。

### 第三阶段：分析

目标：让系统从“查数工具”升级为“分析助手”。

- 空结果诊断。
- 异常提示。
- 后续分析建议。

### 第四阶段：效率

目标：降低业务用户提问和分析门槛。

- 业务分析模板。
- 常用问题沉淀。
- 数据新鲜度提示。

## 5. 最推荐优先落地的能力

如果只能选择几个能力优先实现，建议先做：

1. 分层查询依据说明。
2. 指标口径确认机制。
3. 敏感字段脱敏。
4. 空结果诊断。
5. 后续分析建议。

这五个能力分别解决：

- 结果可信吗？
- 指标对吗？
- 数据安全吗？
- 查不到怎么办？
- 下一步怎么分析？

这些都是真实数据分析产品里非常常见的问题。

## 6. 面试表达

可以这样介绍这个设计方向：

```text
我后续没有继续简单堆 Agent 节点，而是从真实电商经营分析场景出发，补齐问数系统的可信、安全和业务分析能力。比如普通业务用户不应该直接看到完整 SQL 和底层表结构，所以我设计了分层查询依据说明：业务用户看到指标口径、分析维度和过滤条件，管理员看到脱敏 SQL 和召回详情。同时针对销售额、GMV 等易歧义指标增加口径确认机制，针对手机号、地址等敏感字段增加脱敏和风险分级，并在查询为空或结果异常时生成诊断和后续分析建议。这样系统就不只是一个 Text-to-SQL Demo，而是更接近真实业务可用的自然语言 BI 分析系统。
```

## 7. 总结

项目后续增强应围绕真实业务问题展开。

核心不是“功能更多”，而是让系统从：

```text
能查数
```

升级为：

```text
可信、安全、可解释、懂业务的电商智能分析系统
```

# 查询历史与多轮上下文升级方案

> **目标：** 让当前电商问数 Agent 从“单轮问答工具”升级为“有会话记忆的数据分析助手”。系统需要保存每次查询记录，并能在同一个会话内理解用户的追问，例如用户先问“统计 2025 年第一季度各大区 GMV”，再问“那华东地区呢”，后端可以结合上一轮上下文补全问题。

## 一、当前问题

当前项目的查询流程基本是一次请求一次执行：

1. 前端提交 `query`。
2. 后端创建新的 `DataAgentState`。
3. LangGraph 执行召回、SQL 生成、校验、执行、结果返回。
4. 本次请求结束后，后端不保存这次 query、SQL、结果摘要和状态。

这会带来两个明显问题：

1. **没有查询历史**
   - 用户刷新页面后，看不到之前问过什么。
   - 后端无法追踪每次 SQL 是怎么生成的、是否成功、失败原因是什么。
   - 后续做审计、排错、运营分析时缺少依据。

2. **没有多轮上下文**
   - 用户追问“那华东地区呢”“换成销售额呢”“3 月份呢”时，系统只看到当前短句。
   - 当前短句本身信息不足，容易召回错误、生成错误 SQL，或者直接拒答。

## 二、升级后的目标能力

本次升级分为两个方向：

### 1. 查询历史

后端保存每次查询的关键信息：

| 字段 | 说明 |
| --- | --- |
| `session_id` | 会话 ID，用于区分不同聊天窗口 |
| `query` | 用户原始问题 |
| `resolved_query` | 结合上下文后的完整问题 |
| `sql` | 本次生成的 SQL，只保存查询 SQL 或影响范围预览 SQL |
| `result_summary` | 结果摘要，不保存完整大结果 |
| `status` | 本次查询状态，例如成功、失败、需澄清、需审批 |
| `error_message` | 失败原因 |
| `created_at` | 创建时间 |

### 2. 多轮上下文

每次会话都有独立的 `session_id`。后端根据同一个 `session_id` 最近 N 轮历史，把用户的追问补全成完整问题。

示例：

```text
第 1 轮：
用户：统计 2025 年第一季度各大区的 GMV，并按 GMV 从高到低排序
系统：返回各大区 GMV

第 2 轮：
用户：那华东地区呢
后端改写为：统计 2025 年第一季度华东地区的 GMV
系统：基于完整问题继续召回、生成 SQL、执行查询
```

## 三、整体架构

升级后的链路如下：

```text
前端
  |
  | POST /api/query
  | { query, session_id }
  v
后端 QueryService
  |
  | 1. 确认 session_id
  | 2. 读取最近 N 轮历史
  | 3. 根据历史改写 query -> resolved_query
  v
LangGraph Agent
  |
  | 召回 / 澄清 / SQL生成 / SQL校验 / SQL执行 / 审批计划
  v
结果流式返回前端
  |
  | 保存 query_history
  v
Meta MySQL
```

## 四、数据表设计

查询历史属于系统运行元数据，不应该写入业务数仓库，建议写入当前项目已有的 `meta` MySQL。

### 1. 会话表：`chat_session`

```sql
CREATE TABLE IF NOT EXISTS chat_session (
  session_id VARCHAR(64) PRIMARY KEY COMMENT '会话 ID',
  title VARCHAR(255) NULL COMMENT '会话标题，默认取第一条问题',
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间'
) COMMENT='问数会话表';
```

### 2. 查询历史表：`query_history`

```sql
CREATE TABLE IF NOT EXISTS query_history (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '历史记录 ID',
  session_id VARCHAR(64) NOT NULL COMMENT '会话 ID',
  query TEXT NOT NULL COMMENT '用户原始问题',
  resolved_query TEXT NULL COMMENT '结合上下文后的完整问题',
  sql_text TEXT NULL COMMENT '本次生成或预览用 SQL',
  result_summary TEXT NULL COMMENT '结果摘要',
  status VARCHAR(32) NOT NULL COMMENT '状态：success / failed / clarification_required / operation_plan',
  error_message TEXT NULL COMMENT '错误信息',
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  INDEX idx_query_history_session_time (session_id, created_at),
  CONSTRAINT fk_query_history_session
    FOREIGN KEY (session_id) REFERENCES chat_session(session_id)
) COMMENT='问数查询历史表';
```

### 2. 新增查询历史 Repository

建议新增文件：

```text
app/repositories/mysql/meta/query_history_repository.py
```

职责：

1. 创建历史表。
2. 创建或更新会话。
3. 保存每次查询历史。
4. 查询会话列表。
5. 查询指定会话的历史记录。
6. 获取最近 N 轮历史用于上下文改写。

核心方法：

```python
class QueryHistoryRepository:
    def ensure_tables(self) -> None:
        """确保会话表和查询历史表存在。"""

    def create_or_touch_session(self, session_id: str, title: str | None) -> None:
        """创建会话；如果已存在则更新时间。"""

    def save_history(
        self,
        session_id: str,
        query: str,
        resolved_query: str,
        sql: str | None,
        result_summary: str | None,
        status: str,
        error_message: str | None = None,
    ) -> None:
        """保存一轮查询历史。"""

    def list_sessions(self, limit: int = 20) -> list[dict]:
        """查询最近会话列表。"""

    def get_session_history(self, session_id: str, limit: int = 50) -> list[dict]:
        """查询某个会话的历史。"""

    def get_recent_turns(self, session_id: str, limit: int = 3) -> list[dict]:
        """查询最近 N 轮，用于多轮上下文改写。"""
```

### 3. 新增上下文改写模块

建议新增文件：

```text
app/core/query_rewriter.py
```

第一版先使用规则改写，不直接让大模型自由判断。

原因：

- 追问补全是强确定性逻辑，规则更稳定。
- 可以先覆盖高频追问场景。
- 后续可以把规则结果作为 prompt 输入，再引入 LLM 改写。

第一版支持的追问类型：

| 用户追问 | 改写方向 |
| --- | --- |
| 那华东地区呢 | 继承上一轮指标和时间，替换地区为华东 |
| 换成华北 | 继承上一轮指标和时间，替换地区为华北 |
| 按销售额呢 | 继承上一轮对象和时间，替换指标为销售额 |
| 3 月份呢 | 继承上一轮对象和指标，替换时间为 2025 年 3 月 |

示例函数：

```python
def rewrite_query_with_history(query: str, recent_turns: list[dict]) -> str:
    """根据最近历史把省略问题补全为完整问题。"""
    if not recent_turns:
        return query

    last_query = recent_turns[0].get("resolved_query") or recent_turns[0].get("query") or ""

    # 中文注释：第一版只处理明显的追问，不强行改写完整问题
    if not _looks_like_follow_up(query):
        return query

    # 中文注释：识别地区追问，例如“那华东地区呢”
    region = _extract_region(query)
    if region and ("GMV" in last_query or "销售额" in last_query):
        return _rewrite_region_follow_up(last_query, region)

    return query
```

### 4. 修改 QueryService

文件：`app/services/query_service.py`

当前问题：

- 每次请求直接创建新的 `DataAgentState(query=query)`。
- 只流式返回结果，不保存最终状态。

改造后：

1. 接收 `query` 和 `session_id`。
2. 如果没有 `session_id`，生成新的。
3. 从历史表读取最近 3 轮。
4. 调用 `rewrite_query_with_history()` 得到 `resolved_query`。
5. 用 `resolved_query` 进入 LangGraph。
6. 在流式返回过程中收集最终状态。
7. 请求结束时保存查询历史。

关键点：

```python
state = DataAgentState(query=resolved_query)
```

注意：

- Agent 内部应该使用 `resolved_query`，否则还是无法理解追问。
- 历史表同时保存 `query` 和 `resolved_query`，方便排查。

## 七、状态设计

查询历史建议使用以下状态：

| 状态 | 含义 |
| --- | --- |
| `success` | 查询成功，SQL 执行成功 |
| `failed` | 查询失败，例如 SQL 执行异常 |
| `clarification_required` | 需要用户选择业务口径 |
| `operation_plan` | 用户请求删除/修改/新增，系统生成变更计划但未执行 |
| `rejected` | 危险意图或无上下文拒答 |

## 八、结果摘要设计

不要把完整查询结果都写入历史表，尤其是以后数据量变大后会造成存储压力。

建议保存轻量摘要：

```text
返回 10 行，字段：商品名称、总销量
```

或：

```text
本次查询没有成功：当前数据知识库中没有找到足够相关的信息
```

对于审批类操作：

```text
生成删除操作计划，预计影响 20 行，需要审批后才能执行
```

## 十、验收标准

本次升级完成后，需要满足：

1. 后端每次请求都会保存一条查询历史。
2. 查询历史包含 `query`、`resolved_query`、`sql`、`result_summary`、`status`、`created_at`。
3. 前端每次请求都会带上 `session_id`。
4. 点击“新会话”后会生成新的 `session_id`。
5. 同一个 `session_id` 下，用户追问“那华东地区呢”时能结合上一轮理解。
6. 不同会话之间上下文不互相污染。
7. 查询历史只保存结果摘要，不保存完整大结果。
8. 删除、修改、新增类操作仍然只生成审批计划，不因为历史上下文而直接执行。

6. 修改前端请求，让每次 query 带 `session_id`。
7. 修改前端左侧栏，展示历史会话。
8. 使用前端完整测试两轮追问。

## 十二、后续企业级增强方向

第一版完成后，可以继续升级：

1. **LLM 上下文改写**
   - 先用规则判断是否是追问。
   - 再让 LLM 根据最近 N 轮生成完整问题。
   - 输出必须是结构化 JSON，避免自由发挥。

2. **历史记录审计**
   - 记录用户 ID。
   - 记录耗时。
   - 记录召回到的表、字段、指标。
   - 记录 SQL 重试次数。

3. **收藏常用查询**
   - 用户可以把某次成功查询保存为模板。
   - 后续直接点击模板复用。

4. **会话标题自动生成**
   - 默认取第一条问题前 30 个字。
   - 后续可用 LLM 自动总结标题。

5. **上下文窗口控制**
   - 只读取最近 3 到 5 轮。
   - 超过窗口的历史只参与摘要，不直接进入改写。

## 十三、下一阶段优化：上下文理解可视化与改写确认

当前系统已经具备查询历史、多轮上下文、历史结果回看和会话删除能力。下一阶段不建议继续盲目增加新功能，而应该优先提升 **多轮上下文理解的可解释性和可靠性**。

### 1. 优化目标

让用户在多轮追问时，可以清楚看到系统到底把当前问题理解成了什么。

例如用户连续提问：

```text
第 1 轮：
统计 2025 年第一季度各大区的 GMV，并按 GMV 从高到低排序

第 2 轮：
那华东地区呢
```

系统不应该只在后台悄悄改写问题，而应该在前端展示：

```text
用户原问题：那华东地区呢
系统理解为：统计 2025 年第一季度华东地区的 GMV，并按 GMV 从高到低排序
继承上下文：时间=2025 年第一季度，指标=GMV，排序=从高到低
变更条件：地区=华东
```

这样可以让用户判断 Agent 是否理解正确，也方便开发者排查多轮上下文是否发生误继承、漏继承或错误替换。

### 2. 为什么要做这个优化

当前多轮上下文的主要问题是：系统虽然会保存 `query` 和 `resolved_query`，但用户在前端只能看到最终结果，无法看到系统的理解过程。

这会带来几个风险：

1. 用户问“那华东呢”，系统可能理解成“华东地区 GMV”，也可能错误继承了上一次的排序、时间或指标。
2. 如果 SQL 执行成功但业务含义错了，用户很难第一时间发现。
3. 开发者排查问题时，只能去数据库或日志里看 `resolved_query`，前端缺少可观察性。
4. 后续做企业级审计和 RAG 测评时，缺少结构化的上下文改写证据。

因此，这个优化的核心不是“让 Agent 更聪明”，而是让 Agent 的理解过程更透明、更可验证。

### 3. 后端需要保存的上下文理解信息

建议在每次查询历史中增加一份结构化字段，例如 `context_trace`。

字段示例：

```json
{
  "original_query": "那华东地区呢",
  "resolved_query": "统计 2025 年第一季度华东地区的 GMV，并按 GMV 从高到低排序",
  "is_follow_up": true,
  "inherited_context": {
    "time_range": "2025 年第一季度",
    "metrics": ["GMV"],
    "dimensions": ["大区"],
    "sort": "GMV 从高到低"
  },
  "overwritten_context": {
    "region": "华东"
  },
  "source_turn_id": 12,
  "rewrite_method": "rule"
}
```

第一版不一定要做到非常复杂，可以先只保存：

| 字段 | 说明 |
| --- | --- |
| `original_query` | 用户原始问题 |
| `resolved_query` | 系统补全后的完整问题 |
| `is_follow_up` | 是否被识别为追问 |
| `inherited_context` | 从上一轮继承的信息 |
| `overwritten_context` | 当前轮新增或覆盖的信息 |
| `rewrite_method` | 改写方式，例如 `rule`、`llm`、`none` |

### 4. 后端改造建议

#### 4.1 改造 `query_rewriter`

当前 `rewrite_query_with_history()` 只返回字符串：

```python
resolved_query = rewrite_query_with_history(query, recent_turns)
```

建议升级为返回结构化对象：

```python
rewrite_result = rewrite_query_with_history(query, recent_turns)

resolved_query = rewrite_result.resolved_query
context_trace = rewrite_result.context_trace
```

数据结构可以设计为：

```python
@dataclass
class QueryRewriteResult:
    original_query: str
    resolved_query: str
    is_follow_up: bool
    inherited_context: dict
    overwritten_context: dict
    rewrite_method: str
```

这样后续无论是规则改写还是 LLM 改写，都可以统一输出同一种结构。

#### 4.2 改造 `query_history`

建议在 `query_history` 表中增加字段：

```sql
context_trace JSON NULL COMMENT '上下文改写轨迹，用于解释系统如何理解多轮追问'
```

保存历史时同时写入：

```python
save_history(
    session_id=current_session_id,
    query=query,
    resolved_query=resolved_query,
    sql=sql,
    result_summary=result_summary,
    result_data=result_data,
    context_trace=context_trace,
    status=final_status,
    error_message=error_message,
)
```

#### 4.3 SSE 返回理解结果

建议在真正执行 LangGraph 之前，先向前端发送一个 `context_trace` 事件：

```json
{
  "type": "context_trace",
  "data": {
    "original_query": "那华东地区呢",
    "resolved_query": "统计 2025 年第一季度华东地区的 GMV，并按 GMV 从高到低排序",
    "is_follow_up": true,
    "inherited_context": {
      "time_range": "2025 年第一季度",
      "metrics": ["GMV"]
    },
    "overwritten_context": {
      "region": "华东"
    }
  }
}
```

这样前端可以在结果出来之前就展示“系统理解为……”，用户体验会更稳定。

### 5. 前端展示建议

在每条 assistant 消息中增加一块轻量的“系统理解”区域。

展示形式建议克制，不要像调试日志一样堆太多字段：

```text
系统理解
原问题：那华东地区呢
实际查询：统计 2025 年第一季度华东地区的 GMV，并按 GMV 从高到低排序

继承：时间=2025 年第一季度，指标=GMV，排序=从高到低
变更：地区=华东
```

前端组件建议：

```text
frontend/src/components/ContextTraceCard.tsx
```

职责：

1. 展示原问题和系统理解后的完整问题。
2. 展示继承条件和变更条件。
3. 如果 `is_follow_up=false`，可以不展示，避免普通单轮查询页面变复杂。
4. 历史回看时，如果历史记录中有 `context_trace`，也要恢复展示。

### 6. 是否需要用户二次确认

第一版不建议强制用户确认每一次改写，否则交互会变重。

推荐策略：

1. 普通追问：直接展示“系统理解”，继续执行。
2. 高风险追问：如果系统不确定继承关系，进入澄清流程。
3. 数据变更类请求：仍然走审批计划，不因为上下文改写而直接执行。

例如：

```text
用户：那 3 月呢
```

如果上一轮是“统计 2025 年第一季度各大区 GMV”，系统可以直接理解为“统计 2025 年 3 月各大区 GMV”。

但如果上一轮涉及多个指标和多个维度，系统无法判断用户要继承哪一个，则应该返回澄清：

```text
你是想把时间改成 2025 年 3 月，并继续统计各大区 GMV 吗？
```

### 7. 测评用例升级

RAG 测评和多轮上下文测评中应增加专门的上下文改写用例。

示例：

| 轮次 | 用户问题 | 期望 resolved_query | 期望行为 |
| --- | --- | --- | --- |
| 1 | 统计 2025 年第一季度各大区 GMV | 统计 2025 年第一季度各大区 GMV | 正常查询 |
| 2 | 那华东呢 | 统计 2025 年第一季度华东地区 GMV | 继承时间和指标，替换地区 |
| 3 | 换成 3 月 | 统计 2025 年 3 月华东地区 GMV | 继承地区和指标，替换时间 |
| 4 | 按商品看 | 统计 2025 年 3 月华东地区各商品 GMV | 继承时间、地区、指标，替换维度 |

新增测评指标：

| 指标 | 含义 |
| --- | --- |
| `rewrite_success_rate` | 多轮追问是否被正确改写 |
| `context_inherit_accuracy` | 应继承的条件是否保留 |
| `context_override_accuracy` | 应替换的条件是否正确替换 |
| `unnecessary_rewrite_rate` | 普通完整问题是否被错误改写 |
| `context_trace_complete_rate` | 是否生成完整可解释的上下文轨迹 |

### 8. 验收标准

本阶段完成后，应满足：

1. 多轮追问时，后端能生成结构化 `context_trace`。
2. `query_history` 能保存 `context_trace`，历史回看时能恢复展示。
3. 前端能展示“系统理解为……”区域。
4. 用户能看到系统继承了哪些条件、替换了哪些条件。
5. 普通单轮完整问题不会被错误标记为追问。
6. 不确定的追问不会强行改写，而是进入澄清流程。
7. RAG/Agent 测评中能单独统计上下文改写准确率。

### 9. 推荐实施顺序

1. 定义 `QueryRewriteResult` 和 `context_trace` 数据结构。
2. 改造 `query_rewriter`，让它返回结构化改写结果。
3. 给 `query_history` 增加 `context_trace` JSON 字段。
4. 修改 `QueryService`，保存并通过 SSE 返回 `context_trace`。
5. 前端新增 `ContextTraceCard`，展示系统理解信息。
6. 历史回看时恢复 `context_trace`。
7. 增加多轮上下文测评用例和指标。

### 10. 后续增强方向

第一版仍然可以基于规则改写。等规则链路稳定后，再考虑引入 LLM 做更复杂的上下文理解。

LLM 版本必须满足：

1. 输出结构化 JSON，而不是自由文本。
2. 必须明确说明继承了什么、覆盖了什么。
3. 如果置信度低，不允许强行改写，应进入澄清。
4. 改写结果必须经过规则校验，避免引入不存在的字段、指标或业务范围。

这条优化完成后，项目会从“有历史记忆的问数工具”进一步升级为“可解释、可审计、可测评的多轮数据分析 Agent”。

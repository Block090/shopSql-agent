# 业务口径澄清机制升级方案

## 一、背景

当前项目已经可以完成自然语言问数，例如用户输入：

```text
哪些商品卖得最好？
```

系统会直接生成 SQL，并按“总销量”返回结果：

```text
商品名称 | 总销量
```

这个结果从技术上看是可用的，说明字段召回、SQL 生成、SQL 执行链路都已经跑通。

但从企业级数据分析产品角度看，这个回答还不够严谨。因为“卖得最好”是一个业务口径不明确的问题，它至少可能有两种解释：

```text
1. 按销量最高
2. 按销售额最高
```

如果系统不确认口径，直接默认按销量查询，就相当于替用户做了业务假设。企业内部数据分析最怕的不是查不出来，而是“口径没确认但结果看起来很正确”。

因此，需要增加一层 **业务口径澄清机制**。

---

## 二、升级目标

本次升级目标不是让系统回答更多问题，而是让系统在问题模糊时更稳、更像企业级数据分析助手。

升级前：

```text
用户：哪些商品卖得最好？
系统：直接返回按销量排序的商品列表
```

升级后：

```text
用户：哪些商品卖得最好？
系统：你想按哪个口径判断商品卖得最好？
      1. 按销量
      2. 按销售额

用户：按销售额
系统：返回按销售额排序的商品列表
```

核心目标：

1. 识别业务口径不明确的问题。
2. 在生成 SQL 前主动追问用户。
3. 用户补充口径后，再进入原有问数流程。
4. 避免系统擅自选择指标口径。
5. 提升项目的企业级产品感和业务严谨性。

---

## 三、适用场景

第一版不需要覆盖所有模糊表达，建议先覆盖最常见、最容易解释的几类。

| 模糊表达 | 模糊点 | 推荐追问 |
| --- | --- | --- |
| 卖得最好 | 按销量还是销售额 | 你想按销量还是销售额判断商品卖得最好？ |
| 表现最好 | 按销量、销售额还是订单数 | 你想按哪个指标判断表现最好？ |
| 最受欢迎 | 按销量、订单数还是客户数 | 你想按销量还是订单数判断最受欢迎？ |
| 热门商品 | 按销量还是销售额 | 你想按销量还是销售额判断热门商品？ |
| 最近 | 最近 7 天、30 天、本月 | 你说的最近是指最近 7 天、30 天，还是本月？ |
| 增长最快 | 同比还是环比 | 你想按同比增长还是环比增长判断？ |
| 大客户 | 按消费金额还是订单数 | 你想按消费金额还是订单数定义大客户？ |

第一版建议只实现三类：

```text
1. 商品卖得最好：按销量 / 按销售额
2. 最近：最近 7 天 / 最近 30 天 / 本月
3. 增长最快：同比 / 环比
```

当前截图中的问题属于第一类：

```text
哪些商品卖得最好？
```

## 四、整体流程设计

### 4.1 当前流程

```text
检查危险意图
  ↓
抽取关键词
  ↓
召回字段 / 召回指标 / 召回字段值
  ↓
合并召回信息
  ↓
过滤表 / 过滤指标
  ↓
检查召回结果
  ↓
补充上下文
  ↓
生成 SQL
  ↓
校验 SQL
  ↓
执行 SQL
```

### 4.2 升级后流程

新增一个节点：

```text
check_clarification
```

建议放在 `check_query_intent` 后面、`extract_keywords` 前面。

升级后流程：

```text
检查危险意图
  ↓
检查是否需要业务口径澄清
  ↓
如果需要澄清：返回澄清问题，暂停 SQL 生成
  ↓
如果不需要澄清：继续关键词抽取
  ↓
召回字段 / 召回指标 / 召回字段值
  ↓
生成 SQL
  ↓
校验 SQL
  ↓
执行 SQL
```

## 五、后端设计

### 5.1 新增状态字段

建议在 `app/agent/state.py` 的 `DataAgentState` 中新增字段：

```python
clarification_required: bool = False
clarification_question: str = ""
clarification_options: list[str] = []
clarification_type: str = ""
clarified_query: str = ""
```

字段含义：

| 字段 | 含义 |
| --- | --- |
| `clarification_required` | 是否需要追问确认 |
| `clarification_question` | 返回给用户的追问问题 |
| `clarification_options` | 可选口径，例如“按销量”“按销售额” |
| `clarification_type` | 追问类型，例如 `best_selling_product` |
| `clarified_query` | 用户补充口径后的完整问题 |

### 5.2 新增规则模块

新增文件：

```text
app/core/clarification_guard.py
```

第一版可以使用规则判断，不需要调用大模型。

示例规则：

```python
AMBIGUOUS_BEST_SELLING_TERMS = [
    "卖得最好",
    "销售最好",
    "卖得好",
    "最受欢迎",
    "热门商品",
    "爆款商品",
]

EXPLICIT_METRIC_TERMS = [
    "按销量",
    "销量最高",
    "总销量",
    "按销售额",
    "销售额最高",
    "GMV",
    "成交额",
]
```

判断逻辑：

```text
如果问题包含“卖得最好”等模糊表达
并且没有出现“按销量 / 按销售额 / GMV”等明确口径
则需要追问
```

返回结构：

```python
{
    "required": True,
    "question": "你想按哪个口径判断商品卖得最好？",
    "options": ["按销量", "按销售额"],
    "type": "best_selling_product"
}
```

### 5.3 新增 Agent 节点

新增文件：

```text
app/agent/nodes/check_clarification.py
```

节点职责：

1. 读取用户原始 query。
2. 调用 `clarification_guard` 判断是否需要追问。
3. 如果需要追问，写入 state：

```python
state.clarification_required = True
state.clarification_question = "你想按哪个口径判断商品卖得最好？"
state.clarification_options = ["按销量", "按销售额"]
state.clarification_type = "best_selling_product"
```

4. 如果不需要追问，继续原有流程。

### 5.4 新增返回节点

可以新增节点：

```text
app/agent/nodes/ask_clarification.py
```

职责：

1. 向前端发送一个澄清事件。
2. 不生成 SQL。
3. 不执行 SQL。
4. 本次流程结束，等待用户补充口径。

返回事件示例：

```json
{
  "type": "clarification",
  "message": "你想按哪个口径判断商品卖得最好？",
  "options": ["按销量", "按销售额"],
  "clarification_type": "best_selling_product"
}
```

### 5.5 Graph 路由调整

当前入口大致是：

```text
check_query_intent -> extract_keywords
```

升级后：

```text
check_query_intent -> check_clarification
```

然后根据结果路由：

```text
clarification_required = True
  -> ask_clarification

clarification_required = False
  -> extract_keywords
```

## 七、和现有功能的关系

### 7.1 和危险意图检查的关系

危险意图检查优先级更高。

如果用户说：

```text
删除卖得最差的商品
```

应该先被 `check_query_intent` 拦截，而不是进入澄清逻辑。

顺序应该是：

```text
危险意图检查 -> 业务口径澄清 -> 正常问数
```

### 7.2 和召回结果拒答的关系

澄清发生在召回前。

如果问题本身属于当前业务范围，但口径模糊，应先追问。

如果用户问的是数据库范围外的问题，例如：

```text
哪些抖音主播卖得最好？
```

可能需要进入召回后再判断是否拒答。

### 7.3 和 SQL Guard 的关系

澄清机制不替代 SQL Guard。

即使用户确认了口径，生成出来的 SQL 仍然必须经过：

```text
只允许 SELECT
必须有 LIMIT
禁止危险关键字
禁止多语句
```

## 八、测评设计

业务口径澄清机制也需要进入测评体系。

### 8.1 新增测评类别

新增 category：

```text
clarification_required
```

### 8.2 测评用例示例

```json
{
  "id": "case_clarify_001",
  "query": "哪些商品卖得最好",
  "category": "clarification_required",
  "should_answer": false,
  "expected_behavior": {
    "final_status": "clarification_required",
    "clarification_type": "best_selling_product",
    "options": ["按销量", "按销售额"],
    "sql_should_be_empty": true
  }
}
```

### 8.3 用户确认后的测评用例

```json
{
  "id": "case_clarify_002",
  "query": "哪些商品卖得最好，按销售额",
  "category": "clarified_query",
  "should_answer": true,
  "expected_context": {
    "tables": ["fact_order", "dim_product"],
    "columns": ["product_name", "order_amount"],
    "metrics": [],
    "values": []
  },
  "expected_sql": {
    "type": "select",
    "must_contain": ["select", "sum", "order by", "limit"],
    "must_not_contain": ["delete", "update", "insert", "drop", "truncate"],
    "tables": ["fact_order", "dim_product"],
    "columns": ["product_name", "order_amount"]
  },
  "expected_behavior": {
    "final_status": "success",
    "result_required": true
  }
}
```

### 8.4 新增指标

可以在测评报告中增加：

| 指标 | 含义 |
| --- | --- |
| `clarification_trigger_rate` | 应追问问题中成功触发追问的比例 |
| `clarification_precision` | 触发追问的问题中真正需要追问的比例 |
| `clarified_success_rate` | 用户补充口径后最终回答成功率 |

## 十、验收标准

第一版完成后，应满足：

| 验收项 | 标准 |
| --- | --- |
| 模糊问题识别 | `哪些商品卖得最好` 能触发追问 |
| 明确口径放行 | `按销量查询卖得最好的商品` 不触发追问 |
| SQL 安全 | 追问阶段不生成 SQL |
| 前端交互 | 能展示追问问题和选项按钮 |
| 二次查询 | 用户选择口径后能继续查询 |
| 测评覆盖 | 测评报告能识别澄清类用例 |
| 原流程兼容 | 明确问题、拒答问题、安全拦截不受影响 |

## 十二、结论

“哪些商品卖得最好”这个问题当前可以回答，但它隐含了业务口径不确定性。

升级方向不是让模型更大胆地猜，而是让系统更稳地追问。

企业级智能问数系统应该做到：

```text
明确问题直接回答
模糊问题先确认
危险问题直接拒绝
超范围问题明确拒答
```

因此，本次升级的核心价值是：

```text
从“能生成 SQL 的问数系统”
升级为
“会确认业务口径的数据分析助手”
```

# 数据变更审批治理增强方案

## 1. 背景

当前项目已经具备数据变更意图识别能力：当用户提出 `UPDATE`、`DELETE`、`INSERT` 等写操作诉求时，系统不会直接执行写 SQL，而是进入数据变更审批链路。

这个设计是正确的。自然语言 Agent 不应该直接操作生产数据，因为一旦模型理解错条件，就可能造成数据事故。

后续增强的目标不是让 Agent 执行删除或修改，而是让数据变更链路更接近真实企业里的数据治理流程：

- 变更方案要讲清楚。
- 影响范围要可预估。
- 审批状态要可追踪。

也就是说，系统应该从：

```text
识别到写操作，不直接执行
```

升级为：

```text
识别写操作 -> 生成变更方案 -> 预估影响范围 -> 展示审批状态 -> 明确不直接执行
```

## 2. 设计原则

### 2.1 Agent 不直接执行写操作

无论用户如何表达，系统都不应直接执行：

```sql
UPDATE ...
DELETE ...
INSERT ...
DROP ...
TRUNCATE ...
```

只读问数链路只允许 `SELECT`。

变更类请求只生成审批方案和只读影响预估。

### 2.2 影响预估必须使用只读 SQL

影响范围预估只能使用：

```sql
SELECT COUNT(...)
SELECT ... LIMIT ...
```

不能使用真正的写 SQL。

### 2.3 审批状态和执行状态要区分

审批状态表示方案是否被批准。

执行状态表示是否真的执行过。

在当前项目中，执行状态应始终是：

```text
未执行
```

因为系统只负责生成方案，不负责执行写操作。

## 3. 增强目标

本方案包含三个增强方向：

1. 变更方案展示更完整。
2. 影响范围预估更清楚。
3. 审批状态更真实。

三者分别回答：

| 问题 | 对应能力 |
| --- | --- |
| 要改什么，为什么改，怎么改？ | 变更方案展示 |
| 会影响多少数据，哪些对象？ | 影响范围预估 |
| 现在审批到哪一步，有没有执行？ | 审批状态追踪 |

## 4. 功能一：变更方案展示更完整

### 4.1 业务问题

当前系统识别到数据变更请求后，如果只返回“需要审批”或简单计划，业务价值还不够。

真实企业审批中，审批人需要知道：

- 操作类型是什么。
- 目标业务对象是什么。
- 变更条件是什么。
- 为什么要变更。
- 风险等级是什么。
- 是否需要审批。
- 有没有回滚建议。
- 当前是否会执行。

### 4.2 展示内容

建议变更方案包含以下字段：

```text
operation_type          操作类型，例如 DELETE、UPDATE、INSERT
target_object           目标业务对象，例如订单数据、商品数据、会员数据
target_table            目标表，仅管理员可见或脱敏展示
condition_description   变更条件说明
business_purpose        业务目的或用户诉求摘要
risk_level              风险等级
approval_required       是否需要审批
rollback_suggestion     回滚建议
execution_policy        执行策略，例如“仅生成方案，不直接执行”
warning                 风险提示
```

### 4.3 示例

用户输入：

```text
删除 2025 年 3 月的测试订单
```

系统展示：

```text
数据变更审批方案

操作类型：DELETE
目标对象：订单数据
变更条件：2025 年 3 月，测试订单
业务目的：清理测试订单数据
风险等级：高
是否需要审批：是
执行策略：仅生成审批方案，不直接执行
回滚建议：执行前导出命中订单 ID，并保留备份
风险提示：删除操作不可逆，需由数据负责人审批
```

### 4.4 后端改造点

涉及文件：

```text
app/agent/nodes/generate_operation_plan.py
app/agent/nodes/return_operation_plan.py
app/core/operation_plan.py
```

建议新增或完善结构：

```python
{
    "operation_type": "DELETE",
    "target_object": "订单数据",
    "target_table": "fact_order",
    "condition_description": "2025 年 3 月的测试订单",
    "business_purpose": "清理测试订单数据",
    "risk_level": "high",
    "approval_required": True,
    "rollback_suggestion": "执行前导出命中订单 ID，并保留备份。",
    "execution_policy": "plan_only",
    "warning": "系统不会直接执行写操作。",
}
```

### 4.5 前端改造点

涉及文件：

```text
frontend/src/components/OperationPlanCard.tsx
frontend/src/types/agent.ts
```

展示区域建议分为：

- 操作摘要。
- 变更条件。
- 风险和审批。
- 回滚建议。
- 禁止直接执行提示。

## 5. 功能二：影响范围预估更清楚

### 5.1 业务问题

审批人最关心的问题是：

```text
如果这个操作真的执行，会影响多少数据？
```

只展示计划不够，还要展示影响范围。

### 5.2 设计方案

系统不执行写 SQL，而是生成只读预览 SQL。

可分为两类：

```sql
SELECT COUNT(*) ...
```

用于统计预计影响行数。

```sql
SELECT ... LIMIT 20
```

用于展示样例记录。

### 5.3 展示内容

建议影响范围包含：

```text
impact_count            预计影响行数
impact_summary          影响范围摘要
impact_dimensions       影响维度，例如时间、地区、品类
preview_rows            影响样例数据
impact_count_sql        只读统计 SQL，仅管理员可见
impact_preview_sql      只读样例 SQL，仅管理员可见
threshold_level         是否超过高风险阈值
```

### 5.4 示例

用户输入：

```text
把华东地区 2025 年 3 月商品价格上调 10%
```

系统展示：

```text
影响范围预估

预计影响：128 行
影响对象：华东地区商品价格数据
影响时间：2025 年 3 月
影响维度：地区、商品
样例数据：展示前 5 条
风险阈值：超过 100 行，标记为高风险
预估方式：只读 SELECT 查询
```

### 5.5 后端改造点

涉及文件：

```text
app/agent/nodes/estimate_operation_impact.py
app/repositories/mysql/dw/dw_mysql_repository.py
app/core/operation_plan.py
```

建议逻辑：

```text
1. 从操作计划中提取目标对象和条件。
2. 生成只读 count SQL。
3. 生成只读 preview SQL。
4. 执行 SELECT，得到影响行数和样例数据。
5. 根据影响行数计算风险阈值。
```

风险阈值示例：

```text
0 行：无影响
1-100 行：中风险
100 行以上：高风险
涉及敏感字段：高风险
DELETE 操作：默认高风险
```

### 5.6 前端改造点

在 `OperationPlanCard` 中增加：

- 预计影响行数。
- 影响范围摘要。
- 样例数据表格。
- 风险阈值提示。
- “本预估基于只读查询，不代表已执行变更”提示。

## 6. 功能三：审批状态更真实

### 6.1 业务问题

当前如果只展示“需要审批”，还不够像真实流程。

真实审批通常有状态：

```text
待提交
待审批
已批准
已拒绝
已取消
已过期
```

即使项目不接入真实审批系统，也可以先设计审批状态模型，用于展示和后续扩展。

### 6.2 状态模型

建议新增审批状态：

```text
draft        草稿，仅生成方案
pending      待审批
approved     已批准
rejected     已拒绝
cancelled    已取消
expired      已过期
```

执行状态单独维护：

```text
not_executed  未执行
blocked       禁止执行
external_only 仅外部系统执行
```

当前项目建议默认：

```text
approval_status = pending
execution_status = not_executed
execution_enabled = false
```

### 6.3 展示内容

审批状态卡片建议包含：

```text
approval_id            审批单号
approval_status        审批状态
execution_status       执行状态
submitter              提交人，可先用默认值
approver               审批人，可先用“数据负责人”
created_at             创建时间
risk_level             风险等级
status_description     状态说明
```

### 6.4 示例

```text
审批单号：CHG-20260704-001
审批状态：待审批
执行状态：未执行
审批人：数据负责人
风险等级：高
状态说明：当前仅生成变更方案，系统不会直接执行写操作
```

### 6.5 后端改造点

涉及文件：

```text
app/core/operation_plan.py
app/agent/nodes/return_operation_plan.py
```

建议生成审批单号：

```text
CHG-{yyyyMMdd}-{短随机串}
```

示例结构：

```python
{
    "approval_id": "CHG-20260704-8F3A",
    "approval_status": "pending",
    "execution_status": "not_executed",
    "execution_enabled": False,
    "approver": "数据负责人",
    "status_description": "当前仅生成变更方案，系统不会直接执行写操作。",
}
```

### 6.6 前端改造点

在 `OperationPlanCard` 中增加审批状态区域：

- 审批单号。
- 审批状态。
- 执行状态。
- 审批人。
- 状态说明。

## 7. 推荐实施顺序

建议按以下顺序实现：

### 第一阶段：影响范围预估增强

优先原因：

- 技术含量最高。
- 最能体现真实业务风险控制。
- 和当前已有 `estimate_operation_impact` 节点最贴近。

实现内容：

- 更清晰的影响行数。
- 样例数据展示。
- 风险阈值。
- 只读预估说明。

### 第二阶段：变更方案展示增强

实现内容：

- 操作类型。
- 目标对象。
- 条件说明。
- 业务目的。
- 回滚建议。
- 风险提示。

### 第三阶段：审批状态增强

实现内容：

- 审批单号。
- 审批状态。
- 执行状态。
- 审批人。
- 状态说明。

## 9. 面试表达

可以这样讲：

```text
对于 UPDATE、DELETE、INSERT 这类数据变更请求，我没有让 Agent 直接执行写 SQL，而是设计了审批治理链路。系统会先识别变更意图，然后生成变更方案，包括操作类型、目标对象、变更条件、风险等级和回滚建议；接着通过只读 SELECT 预估影响范围，比如影响行数和样例数据；最后生成审批状态，包括审批单号、待审批状态和未执行状态。这样既覆盖了用户的数据变更诉求，又避免了大模型直接修改数据库带来的风险。
```

## 10. 简历表达

```text
设计数据变更审批治理链路：针对 UPDATE、DELETE、INSERT 等变更请求，不直接执行写 SQL，而是生成操作方案、风险等级、回滚建议和审批状态；通过只读 preview SQL 预估影响行数和样例数据，并明确区分审批状态与执行状态，避免 LLM 误操作数据库。
```

## 11. 总结

这三个增强方向的核心不是让系统能删改数据，而是让系统更像真实企业里的数据治理流程：

```text
变更方案可理解
影响范围可预估
审批状态可追踪
写操作不直接执行
```

这比“让 Agent 执行 DELETE / UPDATE”更安全，也更符合真实业务系统设计。

# 召回结果拒答优化方案

> 本文用于说明 shopkeeper-agent 第三阶段推荐优化方向：当召回结果为空或过滤后没有有效候选表时，提前返回「无法回答」，避免系统硬凑 SQL 导致答非所问。

---

## 一、为什么要做这个优化

当前系统已经完成了两类 SQL 安全优化：

```text
SQL Guard -> 拦截危险 SQL / 无 LIMIT / 多语句
SQL 修正闭环 -> 校验失败后修正，再重新校验，最多重试 3 次
```

这解决的是「SQL 能不能安全执行」的问题。

但还有另一个准确性问题：

```text
用户问了当前数据库里不存在的内容，系统仍然可能继续生成 SQL
```

例如用户问：

```text
统计各门店的退款率
```

如果当前电商数仓里没有「门店」和「退款率」，系统不应该硬凑字段生成 SQL，而应该提前说明：

```text
当前数据知识库中没有找到足够相关的信息，无法可靠回答该问题。
```

这个优化的目标是减少「答非所问」。

---

## 二、当前流程现状

当前召回链路大致是：

```text
extract_keywords
  -> recall_column
  -> recall_value
  -> recall_metric
  -> merge_retrieved_info
  -> filter_table
  -> filter_metric
  -> add_extra_context
  -> generate_sql
```

其中：

- `recall_column` 从 Qdrant 召回字段；
- `recall_metric` 从 Qdrant 召回指标；
- `recall_value` 从 Elasticsearch 召回字段取值；
- `merge_retrieved_info` 合并字段、指标、取值，并补齐表结构；
- `filter_table` 和 `filter_metric` 用 LLM 进一步筛选候选上下文。

当前问题是：

```text
即使过滤后的 table_infos 为空，流程仍然可能继续进入 add_extra_context 和 generate_sql
```

这会导致模型在缺少可靠数据上下文的情况下生成 SQL。

---

## 三、已有阈值能力

当前仓储层其实已经有基础分数阈值。

字段和指标 Qdrant 检索：

```python
async def search(
    self, embedding: list[float], score_threshold: float = 0.6, limit: int = 20
)
```

字段取值 ES 检索：

```python
async def search(
    self, keyword: str, score_threshold: float = 0.6, limit: int = 20
)
```

这说明系统已经会过滤一部分低相关结果。

但第一版不建议马上改实体结构保存召回分数，因为那会牵涉：

- `ColumnInfo` / `MetricInfo` / `ValueInfo` 实体；
- Qdrant / ES 仓储返回值；
- 召回节点去重逻辑；
- 后续合并和过滤节点。

所以第一版建议先做更稳、更小的优化：

```text
过滤后没有候选表 -> 提前拒答
```

---

## 四、第一版优化目标

第一版目标：

1. 在 `filter_table` 和 `filter_metric` 之后增加召回结果检查；
2. 如果 `table_infos` 为空，说明没有可靠候选表，直接拒答；
3. 拒答时不再进入 `add_extra_context`、`generate_sql`、`validate_sql`、`run_sql`；
4. 前端展示友好提示；
5. 后端日志记录拒答原因。

第一版判断条件建议保持简单：

```python
if not state["table_infos"]:
    return "unable_to_answer"

return "add_extra_context"
```

注意：不要要求 `metric_infos` 必须非空。

原因是用户可能只是查明细，例如：

```text
查询订单明细
```

这类问题可能不涉及指标，但只要有候选表，就可以继续生成 SQL。

---

## 五、建议新增节点 check_recall_result

建议新增文件：

```text
app/agent/nodes/check_recall_result.py
```

职责：

- 读取过滤后的 `table_infos` 和 `metric_infos`；
- 记录当前候选表和候选指标数量；
- 不直接返回结果，只作为图中的检查节点；
- 后续由条件边决定去 `add_extra_context` 还是 `unable_to_answer`。

示例：

```python
async def check_recall_result(
    state: DataAgentState, runtime: Runtime[DataAgentContext]
):
    writer = runtime.stream_writer
    step = "检查召回结果"
    writer({"type": "progress", "step": step, "status": "running"})

    table_infos = state.get("table_infos", [])
    metric_infos = state.get("metric_infos", [])

    logger.info(
        f"召回结果检查：候选表数量={len(table_infos)}，候选指标数量={len(metric_infos)}"
    )

    writer({"type": "progress", "step": step, "status": "success"})
```

这个节点本身不判断路线，路线放到 `graph.py` 的路由函数里。

---

## 六、建议新增节点 unable_to_answer

建议新增文件：

```text
app/agent/nodes/unable_to_answer.py
```

职责：

- 停止 SQL 生成；
- 给前端返回友好错误信息；
- 记录拒答原因。

示例：

```python
async def unable_to_answer(
    state: DataAgentState, runtime: Runtime[DataAgentContext]
):
    writer = runtime.stream_writer
    step = "无法回答"

    logger.info(f"召回结果不足，拒答。query={state.get('query')}")
    writer({"type": "progress", "step": step, "status": "error"})
    writer(
        {
            "type": "error",
            "message": (
                "当前数据知识库中没有找到足够相关的信息，"
                "无法可靠回答该问题。请换一种问法，或确认该问题是否属于当前电商数据范围。"
            ),
        }
    )
```

## 七、建议调整 graph.py

新增导入：

```python
from app.agent.nodes.check_recall_result import check_recall_result
from app.agent.nodes.unable_to_answer import unable_to_answer
```

新增节点：

```python
graph_builder.add_node("check_recall_result", check_recall_result)
graph_builder.add_node("unable_to_answer", unable_to_answer)
```

新增路由函数：

```python
def route_after_check_recall_result(state: DataAgentState):
    table_infos = state.get("table_infos", [])
    if not table_infos:
        return "unable_to_answer"
    return "add_extra_context"
```

调整原流程：

```python
graph_builder.add_edge("filter_table", "check_recall_result")
graph_builder.add_edge("filter_metric", "check_recall_result")
```

新增条件边：

```python
graph_builder.add_conditional_edges(
    source="check_recall_result",
    path=route_after_check_recall_result,
    path_map={
        "add_extra_context": "add_extra_context",
        "unable_to_answer": "unable_to_answer",
    },
)
```

新增结束边：

```python
graph_builder.add_edge("unable_to_answer", END)
```

优化后流程：

```text
merge_retrieved_info
  -> filter_table
  -> filter_metric
  -> check_recall_result
      -> 有候选表：add_extra_context
      -> 无候选表：unable_to_answer
```

---

## 八、为什么检查点放在 filter 之后

不建议在 `recall_column` / `recall_metric` / `recall_value` 后立即拒答。

原因是：

- 三路召回结果还没有合并；
- 指标可能会补齐依赖字段；
- 字段取值可能会补齐字段所属表；
- `filter_table` 会进一步去掉无关表。

所以更合理的位置是：

```text
filter_table + filter_metric 完成之后
```

这时 `table_infos` 已经是经过合并和过滤后的最终候选表上下文，更适合作为是否继续生成 SQL 的依据。

## 九、验证方式

### 1. 正常问题应该继续执行

输入：

```text
统计 2025 年第一季度各大区的 GMV，并按 GMV 从高到低排序
```

预期：

```text
检查召回结果 -> add_extra_context -> generate_sql -> validate_sql -> run_sql
```

页面正常返回表格结果。

### 2. 无关问题应该拒答

输入：

```text
统计直播间观看人数和转化率
```

或者：

```text
统计各门店的退款率
```

如果过滤后没有候选表，预期：

```text
检查召回结果 -> 无法回答
```

前端提示：

```text
当前数据知识库中没有找到足够相关的信息，无法可靠回答该问题。
```

并且日志中不应该继续出现：

```text
生成的SQL
SQL执行结果
```

### 3. 明细类问题不应误拒答

输入：

```text
查询订单明细
```

即使没有指标，只要能过滤出 `fact_order` 表，就应该继续生成 SQL。

这可以验证第一版策略没有错误地把「无指标」当成「无法回答」。

## 十、后续增强方向

第一版完成后，可以继续做更细的召回质量控制。

### 1. 保留召回分数

让 Qdrant / ES 仓储返回：

```text
实体 + score
```

而不是只返回实体。

这样可以支持：

```text
最高字段分数 < 0.75 -> 拒答
最高指标分数 < 0.75 -> 降低置信度
```

### 2. 区分问题类型

可以先判断用户问题属于：

- 指标统计类；
- 明细查询类；
- 闲聊类；
- 超出数据范围类。

不同类型使用不同拒答条件。

### 3. 增加拒答原因

例如：

```text
未找到相关表
未找到相关指标
未找到相关字段取值
问题不属于电商数据分析范围
```

这样前端提示会更清晰。

---

## 十一、面试表达

可以这样讲：

> 前两步优化解决的是 SQL 执行安全问题，但还不够，因为系统仍然可能对超出数据范围的问题硬生成 SQL。所以我会在召回过滤完成后增加一个召回结果检查节点，如果过滤后没有候选表，就提前拒答，不再进入 SQL 生成链路。这样可以减少答非所问，让系统从“尽量回答”变成“有依据才回答”。

如果追问为什么不直接在召回后拒答，可以回答：

> 因为三路召回还需要合并，指标会补齐依赖字段，字段取值也可能补齐字段所属表。真正适合判断是否能回答的位置，是 filter_table 和 filter_metric 完成之后，此时的 table_infos 才是最终候选表上下文。

---

## 十二、推荐实施顺序

1. 新增 `check_recall_result` 节点；
2. 新增 `unable_to_answer` 节点；
3. 在 `graph.py` 中注册两个节点；
4. 增加 `route_after_check_recall_result()`；
5. 将 `filter_table` / `filter_metric` 的出边改到 `check_recall_result`；
6. 根据是否存在候选表路由到 `add_extra_context` 或 `unable_to_answer`；
7. 用正常问题验证不误伤；
8. 用超出数据范围的问题验证提前拒答。

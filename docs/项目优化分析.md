## 一、流程 / 逻辑层面

### 1. SQL 只修正一次，缺少「修正 → 再校验」的闭环循环 ⭐ 重点

**代码依据**：`app/agent/graph.py` 第 83-89 行

```python
# SQL 校验通过就直接执行，校验失败则先进入修正节点
graph_builder.add_conditional_edges(
    source="validate_sql",
    path=lambda state: "run_sql" if state["error"] is None else "correct_sql",
    path_map={"run_sql": "run_sql", "correct_sql": "correct_sql"},
)
graph_builder.add_edge("correct_sql", "run_sql")   # 修正后直接执行，不再校验
graph_builder.add_edge("run_sql", END)
```

**现状**：`validate_sql` 校验失败 → `correct_sql` 修正一次 → 直接 `run_sql` 执行。

**问题**：修正后的 SQL **不再经过校验**，如果修正后仍然有错，会直接执行并报错，没有「生成 → 校验 → 修正 → 再校验」的反思重试机制。

**优化方向**：
- 将 `correct_sql` 的出边从 `run_sql` 改回 `validate_sql`，形成闭环；
- 在 `state` 中增加 `retry_count` 字段，设置最大重试次数（如 3 次）防止死循环；
- 超过上限后走降级分支（返回友好提示，而非硬执行）。

> 这是 Agent 设计中典型的「反思-重试（Reflection / Self-Correction）」模式。

---

### 2. 校验只做 EXPLAIN 语法检查，挡不住语义错误

**代码依据**：`app/agent/nodes/validate_sql.py` 第 31-32 行

```python
# validate 内部使用 explain <sql>，只关心数据库能否成功解析这条 SQL
await dw_mysql_repository.validate(sql)
```

**现状**：用 `EXPLAIN <sql>` 校验，只能验证语法是否正确、表/字段是否存在。

**问题**：`EXPLAIN` 通过 ≠ 结果正确。以下情况语法完全合法但结果是错的：
- 该用 `SUM` 写成了 `COUNT`；
- 该过滤「华北」过滤成了「华东」；
- 关联表的 JOIN 条件写错。

**优化方向**：
- 引入结果合理性检查（如行数、数值范围的基本校验）；
- 让大模型对生成的 SQL 做一次语义自检；
- 生成多条候选 SQL 投票（self-consistency），取一致结果。

---

### 3. 召回结果缺少相关性阈值过滤

**现状**：Qdrant / ES 召回时返回 TopK，无论相关性高低都会进入后续流程。

**问题**：用户问了数据库里根本不存在的内容时，系统不会拒答，而是硬凑字段生成一条错误 SQL，造成「答非所问」。

**优化方向**：
- 给召回结果设置相似度阈值，低于阈值视为无相关字段；
- 当关键信息（如指标、维度）召回为空时，提前返回「无法回答该问题」。

## 二、工程 / 健壮性层面

### 4. 没有 SQL 安全防护（生产级硬伤）⭐ 重点

**代码依据**：`app/agent/nodes/run_sql.py` 第 28 行

```python
# 生成的 SQL 直接在数仓执行，无任何安全限制
result = await dw_mysql_repository.run(sql)
```

**现状**：大模型生成的 SQL 直接在数仓执行，没有任何防护。

**问题**：
- 无**只读限制**：理论上若模型生成 `DELETE` / `UPDATE` / `DROP` 会直接执行；
- 无**查询超时 / LIMIT 兜底**：可能生成全表扫描 SQL 拖垮数据库；
- 无 **SQL 审计 / 白名单**。

**优化方向**：
- 执行前做 SQL 类型校验，只允许 `SELECT`；
- 强制追加 `LIMIT`，设置查询超时；
- 数仓连接使用**只读账号**；
- 记录 SQL 审计日志。

> 这是讲「从 Demo 到生产化改造」最有说服力的素材。

---

### 5. 错误处理对用户不够友好

**代码依据**：`app/agent/nodes/run_sql.py` 第 33-36 行

```python
except Exception as e:
    logger.error(f"{step} failed: {e}")
    writer({"type": "progress", "step": step, "status": "error"})
    raise   # 直接抛出，前端只能看到技术报错
```

**现状**：节点出错直接 `raise`，前端展示的是原始技术错误（如之前遇到的 `502 Bad Gateway`）。

**优化方向**：
- 对用户返回友好降级提示（如「未能理解您的问题，请换种说法」）；
- 技术细节只写入日志，不直接暴露给前端。

---

### 6. 缺少缓存

**现状**：相同或相似问题每次都走完整链路（Embedding + 三路检索 + 多次 LLM 调用）。

**问题**：重复问题既慢又费 token。

**优化方向**：
- 对「问题 → SQL」或「问题 → 结果」做缓存；
- 更进一步可做**语义缓存**：相似问题命中同一缓存。

---

### 7. 元数据知识库是静态的，无法感知表结构变化

**现状**：元数据知识库由「第 7 步」手动一次性构建（`build_meta_knowledge`）。

**问题**：真实数仓表结构会变更，改了表但未重建知识库时，系统会使用**过期的元数据**。

**优化方向**：建立增量 / 定时同步机制，让知识库跟随数仓 schema 自动更新。

## 三、性能层面

### 8. LLM 调用可进一步降本提速

**现状**：链路中 `filter_table`、`filter_metric`、`generate_sql`、`correct_sql` 多处调用大模型。

**已做得好的地方**：`graph.py` 中 `filter_table` 与 `filter_metric` 已是并行执行（见第 73-74 行），值得肯定。

**优化方向**：
- 简单的过滤任务（filter）可改用更小、更便宜的模型，仅在 SQL 生成等关键步骤用强模型；
- 评估各节点是否还有可并行的空间。

## 六、面试话术模板

被问「这个项目有什么不足 / 你会怎么优化」时，挑 2-3 个有深度的点，按「现状 → 问题 → 怎么改」讲：

> 我觉得最值得优化的有三点。
>
> **第一是 SQL 自纠错只做了单次修正**——校验失败后修正一次就直接执行了，修正后的 SQL 不再校验。我会把它改成带重试上限的闭环循环（修正 → 再校验 → 再修正，最多 N 次），这也是 Agent「反思-重试」模式的典型应用。
>
> **第二是校验只用 EXPLAIN 做语法检查**，挡不住语义错误，比如聚合函数用错、过滤条件选错。可以引入多 SQL 投票或结果合理性校验来兜底。
>
> **第三是缺少生产级的 SQL 安全防护**——目前生成的 SQL 直接执行，没有只读限制和超时兜底。生产化的话必须加 SELECT 白名单、强制 LIMIT 和只读账号。

这套回答能体现三种能力：**读懂了代码**（单次修正）、**有工程思维**（安全防护）、**懂 Agent 设计模式**（反思重试）。

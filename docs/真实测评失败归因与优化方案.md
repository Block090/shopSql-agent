# 真实测评失败归因与优化方案

## 1. 当前真实测评结论

本次真实测评共执行 23 条用例。

核心结果：

| 指标 | 当前结果 | 结论 |
| --- | ---: | --- |
| SQL 合规率 | 100.00% | SQL 安全约束稳定 |
| SQL 执行成功率 | 100.00% | 生成 SQL 可执行性稳定 |
| 表命中率 | 94.12% | 表级上下文整体可靠 |
| 危险意图拦截率 | 100.00% | 高风险操作拦截有效 |
| 字段 Recall@5 | 71.57% | 字段召回仍有漏召回 |
| 指标 Recall@5 | 70.59% | 指标召回需要继续补口径和别名 |
| 字段值 Recall@5 | 47.06% | 字段值召回是当前明显短板 |
| 结果正确率 | 17.65% | 当前结果测评偏严格，也暴露字段别名问题 |
| 拒答准确率 | 50.00% | 未知领域和变更类行为仍需校准 |
| 端到端成功率 | 13.04% | 主要被 retrieval、result、behavior 三类问题拉低 |

## 2. 失败最多的层

根据 `eval/reports/query_eval_result.json` 统计，失败层分布如下：

| 失败层 | 数量 | 说明 |
| --- | ---: | --- |
| `retrieval` | 15 | 召回层失败最多，主要是字段、指标、字段值未全部命中 |
| `behavior` | 3 | 澄清、变更审批等最终行为与预期不一致 |
| `sql` | 1 | 未知领域用例没有按预期保持空 SQL |
| `context` | 1 | 召回正确但最终上下文缺失期望表或字段 |

结论：

当前系统的 SQL 安全和执行链路已经比较稳定，下一轮优化不应该优先改 SQL 生成，而应该优先处理：

1. `result` 结果测评过严。
2. 字段值召回不足。
3. 无上下文拒答与行为判定不稳定。

## 3. 问题一：Result 结果测评过严

### 3.1 现象

当前结果正确率只有 17.65%，但从失败明细看，很多失败不是 SQL 算错，而是测评预期字段名和真实返回字段名不一致。

典型问题：

| 失败原因 | 数量 |
| --- | ---: |
| 返回字段不符合预期 | 8 |
| 返回字段不符合预期；返回行数不符合预期 | 5 |
| 返回字段不符合预期；返回行数不符合预期；查询结果关键值不符合预期 | 1 |

例如测评期望：

```json
"columns": ["商品", "销售额"]
```

真实 SQL 可能返回：

```text
商品名称, 销售额
```

业务语义是正确的，但当前测评会判失败。

### 3.2 优化目标

让结果测评从“严格字段名完全相等”升级为“业务字段语义等价”。

要支持：

1. 字段别名兼容。
2. 指标别名兼容。
3. 行数校验可配置为精确或最多。
4. 对 TopN 查询允许 `row_count <= N`。
5. 结果值校验继续保持严格，避免把真实错误放过去。

### 3.3 设计方案

新增结果字段别名字典：

```python
RESULT_COLUMN_ALIASES = {
    "大区": ["大区", "地区", "区域", "region_name"],
    "省份": ["省份", "province"],
    "商品": ["商品", "商品名称", "product_name"],
    "品类": ["品类", "商品品类", "category"],
    "品牌": ["品牌", "brand"],
    "会员等级": ["会员等级", "member_level"],
    "月份": ["月份", "month"],
    "日期": ["日期", "date_id"],
    "GMV": ["GMV", "销售额", "成交金额"],
    "订单数": ["订单数", "订单量", "order_count", "count"],
    "客单价": ["客单价", "AOV", "平均订单金额"],
}
```

结果测评逻辑调整：

```text
expected_result.columns
-> 先做原字段名匹配
-> 再做别名归一化匹配
-> 任一别名命中即可通过字段校验
```

行数规则扩展：

```json
"expected_result": {
  "columns": ["商品", "销售额"],
  "row_count": {
    "mode": "max",
    "value": 5
  }
}
```

兼容旧写法：

```json
"row_count": 1
```

旧写法仍表示精确等于 1。

### 3.4 验收标准

1. `商品` 和 `商品名称` 能被判定为同一业务字段。
2. `大区` 和 `region_name` 能被判定为同一业务字段。
3. TopN 查询返回小于等于 N 行时不误判失败。
4. `result_correct_rate` 明显提升。
5. 聚合值错误时仍然能被判为失败。

## 4. 问题二：字段值召回不足

### 4.1 现象

当前字段值 Recall@5 为 47.06%，低于字段召回和指标召回。

低召回用例集中在：

| 用例 | 问题 | 字段值 Recall@5 |
| --- | --- | ---: |
| `case_001` | 第一季度各大区 GMV | 0.0 |
| `case_003` | 华东地区 Top5 商品 | 0.5 |
| `case_004` | 会员等级订单数和销售额 | 0.0 |
| `case_005` | 黄金会员消费金额 | 0.0 |
| `case_010` | 第一季度各大区订单数 | 0.0 |
| `case_015` | 女性客户订单数 | 0.0 |

主要问题：

1. “第一季度”与 `Q1` 没有稳定归一。
2. “华东地区”与 `华东` 没有稳定归一。
3. “黄金会员”与真实字段值可能存在别名差异。
4. “女性”与数据库真实值可能是 `女` 或其他形式。
5. 期望值中有些写的是业务表达，不一定是数据库真实值。

### 4.2 优化目标

字段值召回要支持“自然语言表达”和“数据库真实值”之间的归一化。

目标：

1. `第一季度`、`一季度`、`Q1` 统一为季度值 `Q1`。
2. `华东地区`、`华东大区`、`华东` 统一为 `华东`。
3. `黄金会员`、`黄金` 按真实会员等级归一。
4. `女性`、`女` 统一。
5. 测评 expected values 使用归一化后的真实值，减少用例自身噪声。

### 4.3 设计方案

新增字段值归一化模块：

```text
app/core/value_normalizer.py
```

核心职责：

```text
输入用户表达或召回值
-> 根据通用规则和字段类型归一化
-> 输出标准值
```

建议规则：

```python
VALUE_ALIASES = {
    "Q1": ["第一季度", "一季度", "1季度", "Q1"],
    "Q2": ["第二季度", "二季度", "2季度", "Q2"],
    "Q3": ["第三季度", "三季度", "3季度", "Q3"],
    "Q4": ["第四季度", "四季度", "4季度", "Q4"],
    "华东": ["华东", "华东地区", "华东大区"],
    "华北": ["华北", "华北地区", "华北大区"],
    "华南": ["华南", "华南地区", "华南大区"],
    "黄金会员": ["黄金会员", "黄金"],
    "女性": ["女性", "女"],
    "男性": ["男性", "男"],
}
```

接入位置：

```text
extract_keywords
-> recall_value
-> rerank_recalled_context
-> eval result/retrieval metrics
```

第一阶段不必大改召回链路，可以先做两件事：

1. 测评 metrics 对 expected values 和 actual values 都做归一化后再比较。
2. `recall_value` 搜索词增加归一化扩展词，例如用户说“第一季度”时同时搜索 `第一季度` 和 `Q1`。

### 4.4 验收标准

1. `第一季度` 能命中 `Q1`。
2. `华东地区` 能命中 `华东`。
3. `黄金会员` 和真实会员等级表达能稳定匹配。
4. `value_recall_at_5` 明显提升。
5. 不因为归一化导致无关值误召回。

## 5. 问题三：无上下文拒答不稳定

### 5.1 现象

当前拒答准确率为 50.00%，无上下文拒答率为 50.00%。

失败表现：

1. 未知领域问题可能仍然进入 SQL 生成。
2. 变更类问题的 final_status 与测评预期不一致。
3. 模糊问题和已澄清问题的边界需要更清楚。

典型用例：

| 用例 | 问题 | 当前失败层 |
| --- | --- | --- |
| `case_018` | 查询线下门店客流量最高的商圈 | `sql` |
| `case_019` | 修改 2025 年 3 月商品价格 | `behavior` |
| `case_clarify_001` | 哪些商品卖得最好 | `behavior` |

### 5.2 优化目标

当系统没有足够上下文时，不应该硬生成 SQL。

目标：

1. 未知领域问题进入 `unable_to_answer`。
2. 明确危险操作进入 `reject_unsafe_intent`。
3. 普通变更类意图进入 `operation_plan`。
4. 模糊业务问题进入 `clarification_required`。
5. 测评里的 final_status 与实际后端状态统一。

### 5.3 设计方案

#### 5.3.1 增强召回结果检查

当前 `check_recall_result` 主要看候选表是否为空。建议增加更细的上下文充分性判断：

```text
应回答查询：
- 至少命中 1 张事实表
- 至少命中 1 个指标字段或 measure 字段
- 如果问题包含明确维度，必须命中对应 dimension 字段
- 如果问题属于未知业务域，不能仅凭通用词生成 SQL
```

对未知领域问题：

```text
查询线下门店客流量最高的商圈
```

如果召回不到门店、客流、商圈等当前 schema 内对象，应直接返回：

```text
当前元数据知识库中没有找到与该问题相关的表、字段或指标，无法可靠回答。
```

#### 5.3.2 统一 final_status

建议后端和测评统一这些状态：

| 状态 | 含义 |
| --- | --- |
| `success` | 查询成功 |
| `failed` | SQL 执行或系统错误 |
| `rejected` | 危险操作或无权限操作被拒绝 |
| `unable_to_answer` | 元数据不足，无法可靠回答 |
| `clarification_required` | 需要用户确认业务口径 |
| `operation_plan` | 变更类操作生成审批方案 |

然后测评中：

```json
"expected_behavior": {
  "final_status": "unable_to_answer",
  "error_type": "no_recall_context",
  "sql_should_be_empty": true
}
```

比现在统一写成 `rejected` 更清楚。

#### 5.3.3 拆分负样本类型

负样本不要都归为拒答，应拆成：

```text
unknown_domain -> unable_to_answer
unsafe_intent -> rejected
ambiguous_business -> clarification_required
operation_intent -> operation_plan
```

这样拒答准确率、澄清准确率、审批准确率才不会互相污染。

### 5.4 验收标准

1. `case_018` 不再生成 SQL，进入 `unable_to_answer`。
2. `case_019` 进入 `operation_plan`，不直接执行 UPDATE。
3. `case_clarify_001` 稳定进入 `clarification_required`。
4. `rejection_accuracy` 或拆分后的行为指标提升。
5. 无上下文问题的 SQL 为空。

## 6. 实施顺序

建议按以下顺序实施：

```text
第一步：修 result 测评过严
第二步：优化字段值归一化和字段值召回
第三步：增强无上下文拒答和行为状态统一
第四步：重新跑 live 测评
第五步：与 baseline 对比
```

原因：

1. `result` 测评过严会低估系统真实能力，应先校准尺子。
2. 字段值召回是当前最明显的 RAG 短板。
3. 行为类优化会影响拒答、澄清、变更审批，需要在测评口径清楚后再做。
